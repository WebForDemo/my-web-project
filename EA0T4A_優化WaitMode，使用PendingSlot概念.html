什麼是 LUT？

LUT（Look-Up Table）是 hwtcon 硬體的波形執行槽，共約 128 個。每個 LUT 代表一個正在執行的波形任務。硬體允許多個 LUT 同時執行。

什麼是 Completion（回執）？

可以理解為「收到確認才繼續」的機制：

DRM 送出幀
   → 等 hwtcon 回傳 Completion
   → 收到才能送下一幀

問題點：原本 update_completion
   要等波形真正跑完才 signal
   → DRM 被鎖住 ~250ms
後面提到「Signal Completion」＝ 主動提前傳回執給 DRM，讓它不用等波形跑完就能繼續送幀。

3殘影根因：多個 LUT 同時執行

問題行為

系統在 AUTO（GC16）模式下，允許多個 LUT 同時執行。拖動 Seekbar 時，每幀都啟動一個新 LUT，導致十幾個 LUT 同時對同一區域的像素發送波形。

電壓殘留原理：E-Ink 像素改變顏色需要特定電壓序列。多個波形同時驅動同一像素，電壓互相疊加、殘留，像素無法完整回到目標狀態 → 舊影像留在螢幕上。

為什麼 GC16 特別嚴重？

GC16 需要 330ms，拖動時每 16ms 就有新幀，會同時積累 20 個以上的 LUT 在跑，重疊最嚴重。

4第一版修法與衍生問題

修法思路：Pending Slot（暫存槽）

改用 DU 波形（250ms），並實作 Pending Slot：

  • 波形執行中（BUSY）→ 新幀存入暫存槽，只保留最新幀
  • 波形結束（IDLE）→ 取出暫存槽的幀，送去執行
  • 同一時間最多只有 1 個波形在跑 → 殘影解決 ✓
殘影問題：已解決 ✓

衍生問題：間歇性卡頓

第一版實作有 bug：

  • BUSY 路徑 → 有提前 signal completion ✓
  • IDLE 路徑（每週期第一幀）→ 忘了提前 signal ✗
結果：每當 DU 波形剛跑完（面板回到 IDLE），下一幀進來走 IDLE 路徑，DRM 遇到同步幀就等 update_completion → 凍住 250ms → 使用者感受到「滑一滑卡一下」

卡頓模式說明

DRM 的幀分兩種：

快速幀（大多數）
  只等 mdp_completion（~0.5ms）
  → 不凍住，正常

同步幀（特定條件觸發）
  還要等 update_completion
  → 要等波形跑完（~250ms）
  → DRM 凍住 250ms

使用者感受：
滑動──────卡250ms────滑動──卡250ms──
          ↑                    ↑
      每次波形完成後         下次波形完成後
      第一幀遇到同步幀       第一幀遇到同步幀
關鍵：不是每幀都凍，只有波形剛結束後的「第一幀」可能遇到同步幀，所以是「時而卡頓」而不是「一直卡頓」。

5完整修法：兩個問題一起解決

A
解決殘影：Pending Slot 波形序列化
面板 BUSY 時，新幀存入暫存槽（只保最新），等波形結束後再取出執行。
→ 同一時間只有 1 個波形執行，不重疊，不殘影。
B
解決卡頓：Completion 提前 Signal（Bug 修復）
將「提前 signal 三個 completion」移到判斷 idle/busy 之前。
→ 不管面板是 IDLE 還是 BUSY，DRM 收到幀後立刻得到回執，永遠不等波形。
→ DRM 全速運行，UI 不凍。E-Ink 非同步慢慢更新，互不阻塞。
C
額外優化：防止不必要的電源循環
hwtcon_core_handle_clock_disable()：若暫存槽有待執行的任務，不提前關閉 MMSYS/FITI 電源，避免不必要的硬體重啟延遲。
最終效果：Android UI 60fps 全速跑（DRM 不等波形），E-Ink 螢幕每 250ms 非同步更新一次。快速又無殘影。

5核心邏輯（技術細節）

hwtcon_core_submit_task_internal()  ← 每幀提交

if (此幀是 DU / A2 模式) {

  // ★ 關鍵：三個 completion 全部提前 signal
  // 不管 IDLE/BUSY，DRM 幀進來就立刻拿到回執
  complete(&pmarker->mdp_completion);
  complete(&pmarker->submit_completion);
  complete(&pmarker->update_completion);  // ← 這個最重要

  if (面板 BUSY) {
    // Pending Slot：只保最新幀
    pending_task = 當前幀;
  } else {
    // IDLE：直接啟動波形
    insert_to_wait_for_mdp_list(task);
    wake_up(mdp_trigger_wait_queue);
  }

} else {
  // GC16/AUTO：走原本邏輯，不動
  insert_to_wait_for_mdp_list(task);
  wake_up(mdp_trigger_wait_queue);
}

三個 Completion 說明

mdp_completion
DRM 確認幀已進入 MDP（影像處理器）pipeline 的回執。
每幀都會等 ｜ 原本：MDP 接受後 ~0.5ms signal → 不影響速度
submit_completion
DRM 確認幀已被提交給硬體的回執。
部分幀會等 ｜ 原本：提交後 signal，時間短
update_completion ← 問題根源
DRM 確認螢幕已更新完成的回執。「同步幀」（有 PLANE_PROP_WAIT_COMPLETE 旗標的幀）會等這個。
修復前：等 E-Ink 波形真正跑完才 signal → DRM 凍住 ~250ms
修復後：幀進來就立刻 signal → DRM 繼續跑，E-Ink 非同步更新
Bug：第一版只在 BUSY 路徑 signal，IDLE 路徑（每 DU 週期第一幀）沒 signal → 週期性凍住。修復：提到 if/else 前面，兩條路都覆蓋。

◎Wait Mode vs Pending Slot：概念差在哪？

🟡 Wait Mode（等待模式）

DRM 每送一幀後，強制等 E-Ink 波形跑完（~250ms），才能送下一幀。
E-Ink 的慢速直接傳染給 UI 層。

時間軸 →（每格 ≈ 50ms）
DRM：
f1
⏳ 等待波形…
f2
⏳ 等待波形…
f3
E-Ink：
波形 f1（250ms）
波形 f2（250ms）
✅ 波形不重疊 → 無殘影
❌ DRM 每幀等 250ms → UI 感覺嚴重卡頓
500ms 內 E-Ink 只更新 2 次，DRM 也只送 2 幀
✅ Pending Slot（暫存槽模式）

DRM 送幀後立刻拿到回執，繼續全速送幀。
E-Ink 有空時，從暫存槽取最新那一幀來執行。

f7→f8→f9→f10…
E-Ink：
波形 f1（250ms）
波形 f5（250ms）
✅ 波形不重疊 → 無殘影
✅ DRM 不等 E-Ink → UI 全速 60fps 不卡頓
500ms 內 DRM 送 ~30 幀，E-Ink 執行 2 次（取最新位置）
一句話區別：Wait Mode 讓 DRM 跟著 E-Ink 慢；Pending Slot 讓 E-Ink 自己排隊，DRM 繼續跑。螢幕結果相同（都序列化），但 UI 體驗天差地別。
