📌 一句話定位

TIPTOP 有一套鼎新官方的 Services Gateway 對外整合框架，用 XML 命令進出。我們的 AI 小助手可以當成一個新的呼叫方——初期用來查資料（讀）、長期用來建單（寫）。管道、協定、實測都已確認；還沒定的是「有哪些服務可用」與「呼叫權限」。

🗺️ 現況：地端系統怎麼串（TIPTOP 在哪）
先看今天系統的樣子——單據經簽核系統進到 TIPTOP、再讀寫各模組。這是 AI 要接上去的那套系統。

🗺️ 這是現況：員工的單據（核價、請購、簽核）經 NaNaWeb → EasyFlow 進到 TIPTOP，再讀寫各模組。AI 要接的是 TIPTOP 對外的 Services Gateway（XML Web Service）——另一個入口，不經 EasyFlow 簽核（AI 怎麼接見下方）。

🔄 一來一回：① AI 送出 Request XML → ② TIPTOP 回 Response XML → AI 翻成白話答你。這是鼎新官方的 TIPTOP Services Gateway 架構，本來就設計成可讓外部系統呼叫（AI 就是新的一個呼叫方）。

🔎 Gateway 內部怎麼運作（鼎新官方架構）
鼎新官方把這套叫 TIPTOP Services Gateway：呼叫方指名要哪個服務，Gateway 查登記簿、轉給對應的功能。AI 就是圖上新的一個「Request Service」。

白話：Gateway＝總機（單一入口）；Request Service＝呼叫方指名要哪個服務（AI 就是新的一個）；Services Register＝服務登記簿——跟 IT 要的「完整服務清單」就是它；Service Function＝實際功能（信用檢核、UpdateWOQty…）；Globals／Library＝共用設定與程式庫。

🧭 系統全貌：BPM 與 TIPTOP 的分工
釐清一個常見誤會——不是每件事都在 TIPTOP。
🗂️ BPM（簽核流程）

表單與簽核平台：請假、差旅、客戶基本資料、BOM 申請… 大多在這裡跑完。你的請假在這，不是 TIPTOP。

🏭 TIPTOP（ERP 後端）

營運骨幹：訂單、採購、庫存、生產、財務。業務／生管／財會在用；一般員工少直接碰。

🔗 兩者關係

少數單子（客戶資料、議價、備料…）由 BPM 送進 TIPTOP，走的就是 Services Gateway。純人事單（請假）不進 TIPTOP。

鼎新官方規格：每次呼叫都是一個固定的 XML 信封。最關鍵的是 RequestContent 分兩種——Parameter＝查詢（讀）、Document＝建單（寫），剛好對應我們的兩階段導入。下面用官方骨架＋真實範例對照。
⚙️ 讀 · Parameter（初期）📝 寫 · Document（長期）

① 完整 XML 信封（官方 Request / Response）

通用格式
Request 送出
<Request>
 <Access>                       <!-- 存取訊息 -->
  <Authentication user="tiptop" password="tiptop"/>
  <Connection application="my_app" source="192.168.1.2"/>
  <Organization name="my_organization"/>
  <Locale language="zh_tw"/>
 </Access>
 <RequestContent>
  <Parameter>             <!-- 參數(e.x.自動取號) -->
   <Record>
    <Field name="condition" value="ima01 like '%'"/>
   </Record>
  </Parameter>
  <Document>              <!-- 單據(e.x.建立客戶) -->
   <RecordSet id="1">
    <Master name="my_name">
     <Record><Field name="my_column" value="my_value"/></Record>
    </Master>
    <Detail name="my_name">
     <Record><Field name="my_column" value="my_value"/></Record>
    </Detail>
   </RecordSet>
  </Document>
 </RequestContent>
</Request>
▼ RESPONSE 回傳
<Response>
 <Execution>                  <!-- 服務執行結果 -->
  <Status code="0" sqlcode="0" description=""/>
 </Execution>
 <ResponseContent>
  <Parameter>            <!-- 參數回傳(取號結果) -->
   <Record><Field name="my_parameter" value="my_value"/></Record>
  </Parameter>
  <Document>             <!-- 單據回傳 -->
   <RecordSet id="1"> …Master / Detail… </RecordSet>
  </Document>
 </ResponseContent>
</Response>
<Access>我是誰：帳號、呼叫端代號、營運中心、語系
<RequestContent>我要幹嘛
<Parameter>查詢（讀）→ 初期助手
<Document>建單（寫）→ 長期建單
<Status>執行結果，code=0 代表成功

💡 信封是固定的，換的只是裡面 Parameter / Document 的內容。AI 要做的，就是照這格式把問題組成 XML。

② Access — 表明「我是誰」

身分
Access
<Access>
  <Authentication user="tiptop" password=""/>
  <Connection application="AI_ASSISTANT"
              source="192.168.x.x"/>
  <Organization name="NX"/>
  <Locale language="zh_tw"/>
</Access>
user / password使用者帳密（官方：密碼目前可先填空值）
application呼叫端系統代號 → AI 就填自己的代號
source呼叫端來源 IP / 主機
Organization存取的 ERP 營運中心（NX）
Locale語系（zh_tw / zh_cn / en_us）

💡 application + source 就是「新系統表明身分」的欄位——代表這套架構本來就預期會有不同呼叫方，AI 多接一個門檻不高（是否需先加白名單待 IT 確認）。

③ Parameter — 查詢（讀）· 初期要用的

⚙️ 讀 · 低風險
真實範例：CheckCredit_Shipping
// 送出：只帶一個出貨單號
<Parameter><Record>
  <Field name="oga01" value="N589-26070002"/>
</Record></Parameter>
▼ 回傳：客戶信用全貌
<Parameter><Record>
  <Field name="occ02" value="Crea"/>   客戶
  <Field name="stot"  value="500000"/> 額度
  <Field name="sbal"  value="-525585"/> 餘額(超額)
  <Field name="result" value="N"/>
</Record></Parameter>
條件進官方範例甚至可帶類 SQL 條件，如 ima01 like '%'（查料件）
值出回一組 name／value 欄位（額度、餘額…）
參數/欄位依不同服務各自定義——需向 IT 拿各服務規格

💡 這就是初期 AI 查詢助手要走的模式：白話 → 組 Parameter → 拿值 → 翻白話。log 裡 94 次都是這種讀取。

④ Document — 建單（寫）· 長期才做

📝 寫 · 高風險
單據頭 / 身結構（以核價單為例）
<Document><RecordSet id="1">
  <Master name="核價單頭">
    <Record><Field name="pmc03" value="客戶"/>…</Record>
  </Master>
  <Detail name="核價單身">
    <Record><Field name="pmj03" value="料號"/>…</Record>
  </Detail>
</RecordSet></Document>
Master單據頭（客戶、日期、幣別…）
Detail單據身（每一項料號、數量、單價…）
RecordSet可有多組（頭身可擴充）

💡 長期「對話後自動建單」就是組這種 Document。因為會寫進 ERP，一律「AI 草擬 → 人工確認 → 才送出」，不自動入單。

⑤ Status — 怎麼判斷成功/失敗

結果
Execution
<Execution>
  <Status code="0" sqlcode="0" description=""/>
</Execution>
code錯誤代碼，非 0 代表發生錯誤
sqlcode若有 SQL 錯誤，記錄 ERP SQL error code
description訊息說明

💡 對照 log：多數回傳 code 空白／0＝成功；有一筆 code=100 "Sel oep Null"＝查無資料。

✅ 已從官方規格確認

  • XML 信封格式（Request／Response）
  • 呼叫方怎麼表明身分（application／source）
  • 有查詢（Parameter）與建單（Document）兩種模式
  • 成功／錯誤怎麼判斷（Status code）

❓ 還要向 IT 拿

  • 各服務各自的參數名稱與回傳欄位定義
  • condition 能多彈性（能否自由帶查詢條件）
  • 哪些查詢服務是現成的（尤其庫存/備料）

關鍵缺口

⚠️ 動工前必須釐清的兩點

📦

缺「庫存／備料查詢服務」

目前只確認信用檢核、工單等服務；你最想做的庫存查詢，尚未看到現成服務。協定本身支援（官方範例就是用條件查料件 ima01），差在這支服務有沒有被註冊——需 IT 確認 Services Register，沒有就請鼎新新增。

🔑

呼叫權限未定

AI 當新呼叫方，要能用一個新的 application＋來源 IP 呼叫 Gateway，可能需 IT 先加白名單；還需測試帳號與測試環境，才不會動到正式資料。

建議做法

💡 第一個 PoC 實際長怎樣

🧩 實例：使用者問「合勤現在還能出貨嗎？」→ AI 組 CheckCredit 命令查 TIPTOP → 回「已超額約 52 萬，建議先催收或簽核放行」。第一版只接信用檢核（綠色磚 ✅），其餘查詢等 IT 開服務後同一套接法擴充。

手上已有一支「會動、且有完整實際範例」的服務＝信用檢核（log 連 XML 進出都齊了）。建議第一版就拿它開刀：做「問客戶信用額度／能不能出貨」的對話查詢，把整條 loop 跑通、展示給老闆看；同時平行向 IT 要「庫存查詢服務」。等它到位，同一套 loop 換個服務就能擴到你要的庫存情境——不會卡在等新服務。
落地原則

🔒 資料怎麼落地

TIPTOP 資料屬公司營運核心，AI 一律走私有／內網處理：只讀先行、寫入一律人工確認、呼叫留稽核紀錄、最小權限服務帳號；不把 ERP 資料送到公開的外部服務。

回報前 · 對 IT 必問

📋 待 IT 確認清單

  • 服務清單：Services Register 完整清單、標讀/寫——有沒有現成的「庫存／備料查詢」服務？
  • 服務規格：目標查詢服務要帶哪些 Parameter、會回哪些 Field？
  • 查詢彈性：condition 能多彈性，能不能自由帶查詢條件？
  • 呼叫權限：用新的 application＋來源 IP 呼叫，需不需要先加白名單？有沒有測試帳號？
  • 測試環境：在哪試打，不會動到正式資料？
  • 新增服務：若要新開查詢服務，走什麼流程、由誰做（鼎新 or IT）？

👉 建議下一步

1. 建 PoC（信用檢核）：把「白話→XML→呼叫→翻白話」整條跑通。
2. 平行向 IT 要：庫存查詢服務 ＋ Services Register 清單 ＋ 測試權限。
3. 跑通且服務到位後，擴到庫存等情境，再評估長期「自動建單」。
