01 / 07 起因

換了開機動畫,卻在最後閃出舊品牌圖

更新開機動畫後,動畫播完、進桌面前,會閃一張舊的「mobiscribe / Loading…」畫面,與新動畫不連貫。

觀察到的現象

動畫 → ？ → 桌面

新動畫正常播放,但結尾多了一張舊品牌圖才跳桌面,在電子紙上「定格」得很明顯。

追查結果

那是「過場畫面」

它不是動畫的一部分,而是 frameworks/base 一筆客製 patch 放的過場底圖,沿用了舊動畫的最後一張。

本質　過場圖原設計 = 動畫最後一張,用來無縫銜接。動畫換新後它沒跟著換,才會露出舊圖。
02 / 07 原理

為什麼需要這張過場畫面

開機動畫結束的那一刻,真正的桌面(launcher)還沒準備好。中間有一段「空窗期」,若不處理,畫面會閃黑或顯示殘影。

① 有時間差

系統開機完成 → 動畫被結束 → 但 launcher 還要時間載入。中間沒有東西可顯示。

② 電子紙放大問題

e-ink 是整頁刷新且會殘留上一張。空窗期容易出現黑屏或殘影,看起來像當機。

③ 過場補洞

用一張品牌靜圖把空窗期填滿,讓「動畫 → 靜圖 → 桌面」順暢,使用者無感。

關鍵角色　Android 內建一個「備援桌面」SystemUserHomeActivity(優先序最低的 HOME)。客製 patch 把它改造成只顯示一張全螢幕品牌圖,正好拿來當這段過場。

04 / 07 過場畫面是誰畫的

SystemUserHomeActivity — Android 的備援桌面

它是 AOSP 內建、優先序最低的 HOME。當系統需要桌面、但真正的 launcher 還沒就緒時,系統會先落到它身上——正好涵蓋動畫結束後那一段。

// core/res/AndroidManifest.xml — 它就是個 HOME，但優先序 -100（最低） <activity android:name="...SystemUserHomeActivity" android:systemUserOnly="true" ...> <intent-filter android:priority="-100"> <action android:name="android.intent.action.MAIN" /> <category android:name="android.intent.category.HOME" />
  • 原本 — AOSP 預設它是近乎空白的備援畫面。
  • 客製後 — 版面改成一張全螢幕品牌底圖(ntx_boot_background.png),拿來當開機過場。
  • 所以 — 動畫結束 → 系統暫時落到這個備援 HOME(顯示品牌圖)→ nHome 就緒後接手。
05 / 07 之前的 patch 改法

commit「Modify the transition screen during startup」

這筆 frameworks/base 的客製 patch 共動了 5 處,把備援桌面改造成乾淨的品牌過場,並順手把開機提示框改成電子紙白底。

res/layout/system_user_home.xml版面砍成全螢幕 FrameLayout,背景 = @drawable/ntx_boot_background(移除文字/進度條)
res/drawable-nodpi/ntx_boot_background.png新增的過場底圖(當時 = 舊動畫最後一張)
app/SystemUserHomeActivity.java加 immersive 全螢幕旗標,隱藏狀態列/導覽列,讓品牌圖滿版
core/res/AndroidManifest.xml主題 Translucent → DeviceDefault.NoActionBar,讓底圖不透明、無標題列
PhoneWindowManager.java開機提示框改白底黑字、進度條黑色(配合電子紙閱讀)
設計意圖　過場圖刻意設成「= 動畫最後一張」,讓動畫播完像定格再進桌面,銜接無縫。
06 / 07 本次修正

把過場圖同步成新動畫

動畫換新後,過場圖還停在舊動畫的最後一張,才會露出舊品牌圖。修法就是讓兩者重新對齊。

1更新開機動畫新的 6 張取代 bootanimation.zip(1860×2480、Stored 不壓縮)。
2過場圖換成「新動畫的最後一張」把 ntx_boot_background.png 換成新動畫第 6 張(同為 1860×2480 PNG,直接替換)。
3重 build system兩者都在 system 分區,一次 build 一起生效。
結果　開機:新動畫 → 定格在新動畫最後一張 → 進桌面,不再出現舊的 mobiscribe Loading,銜接回到無縫。
