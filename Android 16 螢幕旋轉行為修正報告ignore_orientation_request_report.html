📋 執行摘要

Android 16 針對大螢幕（tablet / e-reader）新增了一個政策：系統預設會忽略 App 要求的螢幕方向，導致旋轉失控。 我們的 EA0T4A 產品不需要這個行為，因此透過 產品客製化 Overlay 在 source code 層將預設值改為關閉， Build 後即永久生效，重開機後不會還原。

同時修正了一個因資源宣告缺失導致的 Build 編譯錯誤，確保 Build 能正常完成。

1 問題描述

🔴 現象

設備每次重開機後，App 的螢幕方向設定會被系統忽略，旋轉行為不正常。手動執行 adb shell wm set-ignore-orientation-request false 可暫時修復，但重開機後又會回復。

驗證指令

$ adb shell wm get-ignore-orientation-request
ignoreOrientationRequest true for displayId=0
↑ 開機後預設為 true，系統忽略 App 的旋轉請求，行為異常
2 根本原因分析

⚠️ Android 16 新增大螢幕政策

Android 16 為了改善大螢幕（平板、摺疊機）體驗，新增了 ignoreOrientationRequest 機制：系統預設「忽略」App 要求的螢幕方向，統一由系統決定旋轉。


這個行為是透過 allow_ignore_orientation_request flag 控制，預設值在 AOSP 裡被寫死為 true（開啟），導致 EA0T4A 每次開機都是異常狀態。

→

旋轉行為異常 ❌

同時發現的 Build 錯誤

程式碼引用了 R.bool.config_isWindowManagerIgnoreOrientationRequestAllowed 這個 boolean 資源，但 symbols.xml 缺少對應的 <java-symbol> 宣告，導致 AAPT 沒有將它匯出到 R class，編譯直接失敗。

3 解決方案

✅ 核心策略：透過產品 Overlay 控制預設值

不修改 AOSP 共用邏輯，改在產品 Overlay 層將預設值設為 false，讓 EA0T4A 出廠即關閉此機制。其他產品（如平板）不受影響，AOSP 預設值仍是 true。

修改前 AOSP 預設行為

allow_ignore_orientation_request 的預設值在 Java 裡寫死為 true，所有產品開機都會忽略 App 旋轉請求。

修改後 產品獨立控制

預設值改從 resource 讀取，EA0T4A 透過 Overlay 設成 false，開機即關閉，永久有效，其他產品不受影響。

5 技術說明

為何選擇 Overlay 而非直接改 AOSP source？


1

維護性

AOSP source 修改越少越好，日後 merge 新版本時衝突風險低。Overlay 是 Android 官方提供的產品客製化機制。

2

產品隔離

Overlay 只影響 EA0T4A。若未來有其他產品（如平板）需要開啟此功能，只要不加 Overlay 即可，AOSP 預設值仍是 true。

3

永久生效，不依賴 runtime 指令

過去用 adb shell wm set-ignore-orientation-request false 只是暫時 override，重開機就消失。Overlay 是 build-time 決定，開機後不可被非預期的流程覆蓋。

4

仍保留 server-side 彈性

預設值雖然是 false，但若未來需要臨時測試，仍可透過 device_config put window_manager allow_ignore_orientation_request true 覆蓋，不影響彈性。

6 Build 後驗證方式

步驟一：確認旋轉 flag 是否正確關閉

$ adb shell wm get-ignore-orientation-request
ignoreOrientationRequest false for displayId=0
# ✅ 預期看到 false，表示系統尊重 App 的旋轉請求

步驟二：重開機後確認不會還原

$ adb reboot
# 等待設備重開機完成
$ adb shell wm get-ignore-orientation-request
ignoreOrientationRequest false for displayId=0
# ✅ 重開機後仍是 false，確認修正永久有效
