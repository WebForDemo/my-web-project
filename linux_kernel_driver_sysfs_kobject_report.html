Linux Kernel / Driver / sysfs / kobject 協作報告

從 Android / AOSP 開發者視角理解核心物件模型、驅動註冊流程與 sysfs 對外介面
Kernel Object Model / Driver Core / sysfs / kobject
1 目標與定位

這份報告要回答的不是單一 API 怎麼寫，而是 Linux Kernel、Driver 與 sysfs 如何協作。對 Android/AOSP 開發者來說，很多看似「只是寫一個節點」的行為，背後其實牽涉到 kobject、device model 與 sysfs 的整體結構。

  • Kernel 內部負責維護物件、生命週期、層級與 reference counting
  • Driver 透過 bus / device / driver model 掛入核心框架
  • sysfs 將核心物件以檔案與目錄形式暴露到使用者空間
  • kobject 是三者之間最重要的黏合層

為什麼 AOSP 開發者需要懂這個？

因為在 Android 平台上，很多 HAL、init、SELinux、debug script、factory tool 最後都會碰到像 /sys/class、/sys/devices、/sys/bus 這類節點。你要 debug「節點為什麼不見」、「權限為什麼不對」、「driver 為什麼沒有 bind 上去」，本質上都在處理同一套 object model。

2 先用 CPU 當實際例子

如果你是 Android 開發者，最容易建立直覺的例子就是 CPU 頻率與 governor。因為它同時會碰到實體硬體、kernel 管理、sysfs 節點、framework policy，以及 app 端觸發。

CPU 硬體
→
Kernel / cpufreq driver
→
sysfs
→
HAL / native service
→
Framework
→
App

這條鏈每一層在做什麼

層級 在 CPU 例子中的角色 你通常會看到什麼
實體硬體 真正的 CPU core、cluster、DVFS 能力與時脈控制器 big/little cluster、freq table、電源域
Linux Kernel cpufreq / scheduler / thermal framework 管理 CPU 狀態 policy0、scaling_governor、scaling_cur_freq
Driver SoC 專屬 cpufreq driver 負責把硬體能力掛進 kernel framework probe、freq table、set_target、opp 對應
sysfs 把 CPU policy / governor / min-max freq 對外暴露 /sys/devices/system/cpu/cpufreq/policy0/...
HAL / native 有些平台會由 native daemon 或 vendor service 封裝 sysfs 操作 vendor service、power HAL、thermal HAL
Framework 根據 power mode、interaction、thermal policy 做決策 PowerManager、PerformanceHint、Power HAL 調用
App 不直接碰 CPU driver，而是透過 framework API 觸發策略 遊戲模式、相機啟動、bench app、測試工具

重點理解

App 幾乎不會直接控制 CPU。App 只能表達「我要高效能」、「我要低延遲」；真正把這個需求轉成 governor 或頻率限制的，是 framework 與 kernel 之間的那條鏈。當你在 debug CPU 表現時，不能只看 app，必須一路追到 sysfs 節點與 driver callback。

3 CPU 是怎麼被掛成 sysfs 的

把 CPU 掛成 sysfs，不是「順手輸出一個檔案」那麼簡單。它通常是下面這條流程：

Hardware Description
1
SoC / DT 描述 CPU 與 OPP 能力
device tree 或平台資料先描述有哪些 CPU、支援哪些頻率、電壓與 policy。
↓
Driver Probe
2
cpufreq driver probe
平台 cpufreq driver 註冊到 kernel cpufreq framework，建立 policy 與可支援頻率表。
↓
Kernel Object Model
3
建立 device / kobject / attribute
這一步會把 policy、governor、cur_freq、max_freq 等狀態對應成 sysfs 可見節點。
↓
sysfs Export
4
形成 /sys 路徑
例如 /sys/devices/system/cpu/cpufreq/policy0/scaling_governor 這種節點，背後就是 kernel 物件模型加上 attribute callback 的結果。
↓
Userspace Access
5
HAL / init / debug tool 讀寫
當 userspace 寫入 governor 時，最終還是回到 cpufreq driver 的 callback 去實際改變 CPU 行為。

以常見節點為例

/sys/devices/system/cpu/cpufreq/policy0/scaling_governor /sys/devices/system/cpu/cpufreq/policy0/scaling_cur_freq /sys/devices/system/cpu/cpufreq/policy0/scaling_max_freq /sys/devices/system/cpu/cpufreq/policy0/scaling_min_freq

這些節點不是獨立存在的；它們依賴前面幾步都成功。也就是說，如果 driver 沒 probe、policy 沒建起來、kobject/attribute 沒 export，這些檔案根本不會出現。

你可以怎麼 debug

  • 看 dmesg 確認 cpufreq driver probe 成功沒有
  • 看 /sys/devices/system/cpu/ 下 CPU 與 cpufreq policy 是否存在
  • 如果節點有出來但寫入無效，就回頭追對應 framework 與 driver callback
  • 如果 framework 說已經切模式，但 sysfs 沒變，代表中間層轉換有問題
4 整體架構示意圖
Kernel Core
1
kobject 建立核心物件
每個可管理的核心物件都可以有一個 kobject，它負責名稱、parent-child 關係、sysfs 掛載點與 refcount。
核心概念：name / parent / kset / ktype
↓
Driver Core
2
bus / device / driver 掛入模型
驅動不是單獨存在；它通常會註冊成某個 bus 上的 device 與 driver，並由 driver core 負責 match、probe、remove。
常見物件：struct device / struct device_driver / struct bus_type
↓
sysfs Export
3
sysfs 對外暴露目錄與屬性
核心物件掛上來後，可以透過 attribute group 或 device attribute，形成 /sys 下可讀寫節點，提供 userspace 檢視或控制。
路徑例：/sys/devices / /sys/class / /sys/bus
↓
Userspace
4
init / HAL / app_debug 讀寫節點
Android userspace 不直接理解 kernel object；它只會看到 sysfs 檔案。但這些檔案其實就是後端 driver callback 的入口。
使用方式：cat / echo / property-trigger script / native service
5 kobject 是什麼

kobject 可以把它理解成 Linux kernel 裡「可被管理、可被掛到 sysfs、可被追蹤生命週期」的基本外殼。它不是某個功能驅動本身，而是一個共通的 object infrastructure。

欄位 / 概念 作用 Debug 觀點
name 決定 sysfs 目錄名稱 節點名稱怪異或重複時先看這裡
parent 決定在 sysfs 中掛在哪個父目錄下 路徑不在預期位置時先查 parent 關係
kset 把同類型物件集合管理 class / bus 組織常由 kset 串起
ktype 定義 release 與 default attrs memory leak 或 attrs 缺失時常要追它
refcount 控制生命週期，避免過早釋放 release() 沒被叫或被重複叫都很危險

重要觀念

很多核心結構內部都「內嵌」一個 kobject，例如 struct device。這表示你表面上在操作 device，底下其實同時在使用 kobject 的 reference / sysfs / hierarchy 能力。

struct device { ... struct kobject kobj; ... };
6 Driver 如何接進 Kernel Model

一個 driver 真正「活起來」通常不是只靠 module load，而是經過 driver core 的註冊與匹配流程。簡化後你可以理解為：

driver register
→
device register
→
match
→
probe
  • bus 提供 match 規則，例如 platform、i2c、spi、pci
  • device 代表硬體實體或邏輯裝置
  • driver 提供 probe/remove 與 attribute
  • 一旦 probe 成功，sysfs 路徑、uevent、power management 狀態才會完整建立

對 AOSP 的實際意義

如果你在 Android 看到某個 /sys/class/foo/foo0 節點缺失，不一定是 sysfs 壞了；很可能是 device 根本沒有成功 register，或者 driver 沒有 probe 成功。這時候要先回頭檢查 dmesg、match table、compatible、modalias。

7 sysfs 怎麼來的

sysfs 是 kernel object model 的檔案化呈現。它不是單純把記憶體 dump 成檔案，而是把 object、attribute 與 callback 映射成目錄與節點。

sysfs 路徑 典型來源 常見用途
/sys/devices 實際 device tree / topology 追真正的父子階層與實體裝置
/sys/class class 抽象層 給 userspace 比較穩定的功能路徑
/sys/bus bus framework 看 driver 與 device 是否 bind
/sys/module module 狀態 查參數、refcount、載入情況

attribute 的本質

每個 sysfs 檔案背後通常對應一組 show() / store() callback。使用者執行 cat 時，其實是呼叫 driver 的 show；執行 echo 1 > node 時，其實是呼叫 store。

static ssize_t enable_show(struct device *dev, struct device_attribute *attr, char *buf); static ssize_t enable_store(struct device *dev, struct device_attribute *attr, const char *buf, size_t count); // DEVICE_ATTR_RW(enable) 之後會在 sysfs 形成 enable 節點

一句話總結

kobject 提供物件骨架，driver core 提供註冊與匹配流程，sysfs 提供對外介面。這三個不是平行關係，而是上下疊加的同一條鏈。

9 常見生命週期與 release 問題

理解 kobject_put() 與 release() 很重要，因為很多 bug 不會直接爆在功能點，而是晚一點才在 remove 或 suspend/resume 階段炸掉。

init / register
create sysfs attrs
userspace read/write
unregister / put / release
  • 忘記 put 會造成物件不釋放，sysfs 殘留或記憶體 leak
  • release 沒寫或寫錯，常見 warning 是 object leaked
  • 先刪 attr 再 unregister，順序錯了可能導致 use-after-free
10 Android / AOSP Debug 思路

你看到 sysfs 節點不見時

  • 先查 driver 有沒有 load、probe 有沒有成功
  • 再查 device 是否真的 register 到對應 bus / class
  • 最後才看 sysfs attribute 建立有沒有失敗

你看到節點存在但寫入沒反應時

  • 檢查 store() 是否有被呼叫
  • 確認輸入格式是否被 parser 拒絕
  • 確認是否有權限、SELinux、ro/rw 限制
  • 確認寫入後是否真的改動 driver state，而不是只改到快取變數

你看到 bind / unbind 問題時

直接看：

/sys/bus/<bus>/devices/ /sys/bus/<bus>/drivers/ /sys/class/ /sys/devices/

如果 /sys/class 有節點但 /sys/devices 路徑異常，通常代表 abstraction layer 有建立，但底層 topology 或 symlink 需要再追。

11 實務範例：一個 enable 節點是怎麼形成的

以下是最常見的心智模型：

driver probe()
→
device_create_file()
→
/sys/class/foo/foo0/enable
→
show/store callback
static DEVICE_ATTR_RW(enable); static int foo_probe(struct platform_device *pdev) { int ret; ret = device_create_file(&pdev->dev, &dev_attr_enable); if (ret) return ret; return 0; }

userspace 寫 echo 1 > enable，最後會進到 enable_store()，真正的硬體控制通常就在那裡或它再往下呼叫的函式裡。

12 最後總結

如果只記住一句話：

driver 負責功能，kobject 負責被管理，sysfs 負責被看到。
  • Kernel object model 決定物件怎麼存在
  • Driver core 決定裝置怎麼被註冊、匹配與回收
  • sysfs 決定 userspace 怎麼觀察與控制這些物件
  • 除錯時不要只看節點，要回頭追 object lifecycle 與 driver registration
For Android / AOSP integration and debug: always trace from sysfs path back to device model, not the other way around.
