SeekBar 拖動 E-Ink 快速刷新整合報告

在 Framework 層讓所有滑桿（SeekBar）拖動時自動切換 E-Ink 快速波形、放開後復原
範圍：framework.jar
異動檔案：1 file
機制：ntx.eink_control.GLOBAL_REFRESH 廣播
日期：2026-06-12

▎一句話總結

在 Android 所有滑桿的共同基底類別 AbsSeekBar 加入掛鉤：使用者開始拖動滑桿時，發出廣播要 E-Ink 面板切到「快速波形 + Dither」模式讓滑動順暢；拖動結束時，再發一次廣播把波形恢復成系統預設。只動一個檔案，全系統 SeekBar 自動受惠。

1 先驗知識（背景原理）

▎1-1　E-Ink 螢幕怎麼更新：波形（Waveform）

E-Ink（電子紙）不像 LCD/OLED 可以隨時連續刷新。它每次更新畫面，都要對畫素送出一段電壓序列（波形 Waveform）來翻轉墨水粒子。不同波形在「速度」與「畫質」之間取捨：

波形模式耗時（約）畫質適用場景
GC16 / AUTO~330 ms最好，16 階灰靜態畫面、文字閱讀
DU~250 ms中等，灰階有限一般翻頁
A2 / 快速~120 ms最快，偏黑白連續動作（拖動、滑動）

（數據引用自本專案既有文件 /mnt/aosp2/eink_ghosting_fix_presentation.md，實際數值依面板而定。）

▎1-2　為什麼拖 SeekBar 在 E-Ink 上會「卡 + 殘影」

拖動滑桿是連續、高頻的畫面變化。若維持預設的高畫質慢波形（GC16，每幀 ~330ms）：

  • 單幀更新時間太長，跟不上手指移動 → 明顯卡頓。
  • 多幀波形互相重疊、舊畫面來不及清除 → 滑桿拖出一條殘影尾巴。

解法是：只在拖動的這段期間，把面板切到 A2 類的快速波形（犧牲灰階換速度），手放開後再切回高畫質模式。這正是本次改動要做的事。

▎1-3　NTX 的 E-Ink 控制介面：廣播 ntx.eink_control.*

本平台（NTX / MTK）提供一組 Broadcast Intent 來即時切換 E-Ink 刷新模式。系統服務 PowerManagerService 內部註冊了一個接收器 UpdateModeReceiver，監聽這些 action：

  • ntx.eink_control.GLOBAL_REFRESH　← 本次使用，全螢幕模式切換
  • ntx.eink_control.QUICK_REFRESH
  • ntx.eink_control.SPECIAL_REFRESH / DropFrames

收到後，它會呼叫 Surface.updateAuxRefreshMode(...) 一路下到 native，最終改變面板的刷新行為。任何 App / Framework 程式碼都能用 sendBroadcast() 觸發 —— 這也是原本工程師用 adb shell am broadcast 手動測試的同一條路徑。

這次的工作，本質上就是把「工程師手動下的 adb am broadcast 指令」，改成由 SeekBar 在拖動的正確時機自動發出。
2 需求 / 目標
  • 拖動開始 / 拖動中：發出快速刷新廣播，讓滑桿拖起來順、不留殘影。
    adb shell am broadcast -a ntx.eink_control.GLOBAL_REFRESH --ei updatemode 0 --ei waitmode 0 --ei waveformmode 1 --ei flags 0x2000 --ei dithermode 0x301
  • 拖動結束：發出復原廣播，把波形切回系統預設（高畫質）。
    adb shell am broadcast -a ntx.eink_control.GLOBAL_REFRESH --ei waveformmode 0
  • 要對整機所有 SeekBar 生效（音量、亮度、前光、播放進度條等），而不是改每一個 App。
3 解法：改在 Framework 共同基底類別

Android 中所有滑桿（SeekBar、各家 App 的進度條/音量條）都繼承自同一個基底類別 AbsSeekBar，它位在 AOSP framework。只要在這個類別處理觸控的進入/結束點掛鉤，一處修改 = 全系統生效，不需逐一改 App。

拖動開始
startDrag()
手指真的開始拖動滑桿 → 發「快速波形」廣播
拖動結束 / 取消
onStopTrackingTouch()
放開或被取消 → 發「復原波形」廣播

▎5-1　新增成員（旗標 + 常數）

// NTX e-ink：拖動期間切快速波形，結束後復原 private static final String NTX_EINK_GLOBAL_REFRESH = "ntx.eink_control.GLOBAL_REFRESH"; private boolean mEinkSeekFastRefresh = false;

▎5-2　進入快速波形（對應拖動開始）

private void startEinkSeekFastRefresh() { if (mEinkSeekFastRefresh) return; // 已在快刷狀態，避免重複發送 mEinkSeekFastRefresh = true; final Intent intent = new Intent(NTX_EINK_GLOBAL_REFRESH); intent.putExtra("updatemode", 0); intent.putExtra("waitmode", 0); intent.putExtra("waveformmode", 1); // 快速波形 intent.putExtra("flags", 0x2000); intent.putExtra("dithermode", 0x301); getContext().sendBroadcast(intent); }

▎5-3　復原波形（對應拖動結束）

private void stopEinkSeekFastRefresh() { if (!mEinkSeekFastRefresh) return; // 沒發過 start 就不發 stop mEinkSeekFastRefresh = false; final Intent intent = new Intent(NTX_EINK_GLOBAL_REFRESH); intent.putExtra("waveformmode", 0); // 0 = 回系統預設波形 getContext().sendBroadcast(intent); }

▎5-4　三個掛鉤點

方法時機呼叫
startDrag()真正開始拖動startEinkSeekFastRefresh()
onStopTrackingTouch()放開 / 觸控取消stopEinkSeekFastRefresh()
onDetachedFromWindow()拖動中畫面被移除（保險）stopEinkSeekFastRefresh()
6 廣播參數說明
Extra意義拖動中拖動結束
waveformmode波形模式（速度/畫質取捨）1（快速）0（預設/復原）
updatemode更新模式旗標00（預設）
waitmode是否等待前一幀完成00（預設）
flags面板控制旗標0x20000（預設）
dithermode抖色（Dither）設定，改善快速波形的視覺0x3010（預設）

復原廣播只帶 waveformmode=0，其餘 extra 在接收端 getIntExtra(key, 0) 預設即為 0，行為與 adb 復原指令完全一致。

7 設計考量（為什麼這樣寫）

▎用 mEinkSeekFastRefresh 旗標把關

onStopTrackingTouch() 在「單純點一下定位」時也會被呼叫。用旗標確保：只有真正發過「快速波形」的拖動，結束時才會發「復原」廣播 —— 不會憑空發、也不會漏發，廣播永遠成對。

▎start 掛在 startDrag() 而非 onStartTrackingTouch()

後者在「點擊定位（tap-seek）」時也會被呼叫。掛在 startDrag() 可讓單純點擊不觸發波形切換，避免點一下就閃一次快/慢波形。

▎onDetachedFromWindow() 保險

萬一拖動中畫面被移除、沒收到正常的放開事件，這個保險會補發復原廣播，避免整機面板卡在快速波形（A2）回不來。

相容性佐證：本專案 SystemUI 既有程式碼（PhoneStatusBarView.java）早已用完全相同的寫法發 ntx.eink_control.* 廣播；而接收端 PowerManagerService 讀取的 extra key 名稱與本次使用逐字吻合。因此本改動是沿用既有、已驗證的機制，非新造。
8 影響範圍與注意事項
  • 編譯範圍：改動在 /mnt/aosp2/android16_mtk_system_b0mp5_aiv/frameworks/base，需重新編譯並更新 framework.jar（非單一 App）。
  • 全系統生效：因為改在基底類別，整機所有 SeekBar 拖動都會觸發。若日後只想限定特定 App（例如閱讀器進度條），可再加判斷條件收斂。
  • 廣播時機：僅在「拖動開始」與「拖動結束」各發一次，非每一幀，效能負擔可忽略。
  • 無新權限：使用既有公開廣播機制，未新增 SELinux / 權限需求。
