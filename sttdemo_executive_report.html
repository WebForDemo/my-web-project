sttDemo 離線語音辨識 App 最終說明報告

Executive overview for on-device speech-to-text integration using Android App + JNI + Native Whisper Runtime
專案：sttDemo
定位：Offline On-Device STT Demo App
核心：sean_stt / whisper.cpp
平台：Android + JNI + Native Runtime
日期：2026-05-11
1 專案摘要

▎這個 App 解決什麼問題

把既有的 native 語音辨識能力整合進 Android App，讓裝置可以在無網路情況下完成本地語音轉文字，並同時支援：

  • 麥克風即時辨識
  • 音訊檔辨識（WAV / MP3 / Android 可解碼格式）
  • Debug 對照驗證（Recorded WAV Result）

▎目前可向主管呈現的結論

  • native STT engine 已成功整合到 Android App
  • 檔案辨識路徑已可穩定運作
  • Mic realtime 路徑已打通，並能輸出 Live Result 與 Transcript
  • 系統可明確區分「模型能力」與「realtime pipeline 問題」
2 商業價值

▎離線能力

不依賴雲端服務，不需要網路連線，適合內網、車載、工控、敏感資料場景。

▎可展示性

App 直接提供即時畫面、全文結果與 debug 對照，適合對內展示 PoC 與效能驗證。

▎可延伸性

目前先走 App + JNI，未來若要往 Framework / Service 化發展，可保留 native 核心，只替換上層介面。

這張圖是管理層版本的系統架構圖。重點是：UI 層、Java 控制層、JNI Bridge、Native Runtime、硬體加速路徑的責任劃分，以及 Mic / File / Debug 三條資料流。
4 架構重點
UI 與互動
Java 控制與資料流
JNI 橋接
Native 推論

UI Layer 使用者可見層

包含開始錄音、選取音訊檔、語言選擇，以及畫面上的 Live Result、Transcript、Recorded WAV Result。這一層只負責操作與展示，不直接碰 native 細節。

Java Control Layer 控制核心

RealtimeSttController 負責 AudioRecord、utterance buffer、停頓判斷、realtime 推論節奏、全文累積，以及停止錄音後的 debug WAV compare。

JNI Bridge Layer 邊界封裝

NativeWhisper.java 與 stt_jni.cpp 承接 Java 到 native 的資料傳輸，負責把 float[] 音訊送入引擎，並把結果字串回傳。

Native Runtime Layer 推論引擎

由 libwhisper / sean_stt 執行實際推論，目前裝置上的路徑是 APU Encoder + CPU Decoder Fallback，並處理 UTF-8 字串安全與 stream prompt token 狀態。

5 端到端流程
Mic 路徑

AudioRecord 收進 16kHz / mono / PCM16，Java 控制層轉成 float、累積本句音訊，再觸發 native 轉寫。

→
JNI / Native

Java 將音訊送進 JNI，native 執行 whisper_full()，把結果回傳給 App。

→
畫面更新

App 依穩定度分別更新 Live Result 與 Transcript。

File 路徑

使用者選 WAV / MP3，先由 Java 層解碼成 mono 16k float[]。

→
完整轉寫

直接以 non-streaming 方式做完整推論，這條路目前準確度最高。

→
Debug Compare

Mic 停止後，會把 PCM 另存成 WAV，再走一次檔案辨識，作為品質對照基準。

6 主要模組與職責
檔案 / 模組 層級 責任
SttDemoActivity.java UI 畫面控制、按鈕事件、權限、語言選擇、結果顯示。
RealtimeSttController.java Control 錄音、buffer、停頓切句、realtime 推論節奏、全文累積與 WAV compare。
NativeWhisper.java Bridge Java 對 native 的封裝 API。
NativeWhisperLoader.java Loader 抽出並依序載入 RD 交付的 native libraries。
stt_jni.cpp JNI 建立 whisper context、設定 decoding params、UTF-8 sanitize、stream prompt token state。
AudioFileDecoder.java / WavFileDecoder.java Decode 處理 WAV / MP3 / 壓縮格式解碼並轉成 mono 16k float[]。
WavFileWriter.java Debug 把 Mic PCM16 寫成 last_mic_recording.wav，方便重跑檔案辨識。
7 畫面與使用者語意

▎Live Result

顯示目前正在生成中的即時文字。它的目標是讓使用者看到「當下系統正在組合的內容」，不一定等於最終全文。

▎Transcript

顯示穩定累積下來的全文。當系統判斷某一段已較穩定，或一句話已告一段落，才會正式累積進 Transcript。

▎Recorded WAV Result

這是停止錄音後，把相同 Mic 內容存成 WAV 再做一次完整檔案辨識的結果，用來區分問題是在模型本身，還是在 realtime pipeline。

▎Language Selection

目前支援 Chinese / English / Auto。設定會同時套用到 Mic、File、以及 debug compare 三條路徑。

8 目前成果與判讀

已驗證

  • 模型、.so、JNI、App UI 已完整串通
  • WAV / MP3 檔案辨識可運作
  • Mic 內容可被另存為 WAV 並重跑辨識
  • 可利用對照結果分離「模型問題」與「realtime pipeline 問題」

目前限制

  • realtime mic 品質仍弱於 Recorded WAV Result
  • 目前採用較保守的整句增量重跑策略，延遲會高於真正 streaming STT
  • Auto 對音訊檔通常可用，但對短句 realtime 不一定最穩
9 為什麼這個 PoC 有價值

這個 PoC 已經證明兩件重要的事：

  • 既有 native STT runtime 能成功被 App 使用，不是只能跑 CLI 或單獨部署。
  • 裝置上的辨識品質主要瓶頸已可被定位到 realtime pipeline，而不是整個方案不可行。

換句話說，現在不是在驗證「能不能做」，而是在收斂「怎麼做得更即時、更穩」。

10 建議後續方向

▎短期

繼續優化 realtime 顯示語意與推論節奏，讓 Live Result 更直覺、Transcript 更穩定。

▎中期

評估更貼近 whisper-stream 的 streaming 策略，縮小與檔案辨識結果之間的差距。

▎長期

若產品需要，可往 Service / Framework 方向擴展，但保留現有 native 核心與模型資產。

11 主管講解摘要

這個 App 已經成功把 native Whisper / sean_stt 整合到 Android 端，具備離線麥克風辨識、音訊檔辨識，以及 debug 對照能力。從目前結果來看，模型本身是可用的，系統也具備可展示性。現階段主要工作已從「能否整合」進入「如何把 realtime 體驗做得更好」。
