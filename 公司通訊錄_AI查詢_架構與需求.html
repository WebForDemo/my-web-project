📌 一句話

公司通訊錄的資料源在 BPM 的 SQL（User Table）：由 AD 設定分機、每天同步進 BPM，再由 Portal 的 PhoneSearch 讀出來顯示。我們要在同一份資料上加「自然語言查詢」——等於把軌2「分機查詢助手」從「Excel 寫死、不能更新」升級成「接 BPM SQL、天天同步」。

🤖 AI 小助手怎麼接（PoC 架構）
使用者白話問 → AI 小助手組唯讀查詢 → 連 BPM SQL 的 User Table → 把結果講成白話。跟 ERP 那套同一個模式。

🟢 AI 只透過唯讀帳號連 User Table（不碰其他 BPM 表、只讀不寫）。資料由 AD 每天同步 2 次 → AI 查到「最多半天前」，比 Excel 寫死新鮮太多。

📄

舊：分機助手（Excel 寫死）

資料是上傳的 Excel、靜態不動。人員異動就過期，得人工重做、重傳，常常不準。

🗄️

新：接 BPM SQL（天天同步）

直接讀 BPM SQL；AD 每天同步 2 次進來，AI 查到的是「最多半天前」的資料，免手動維護。

※ 不是即時到秒（若要秒級需直連 AD，成本較高、非本期）；但相較 Excel 寫死已是質的提升。

📋 要做這件事，需要 Scott／MIS 提供

下面 4 項是「AI 能安全連上 BPM SQL、看懂 User Table」的必要材料。取得後即可開始 PoC。

1

PhoneSearch.cs （程式檔）

看現有分機查詢怎麼查、用到哪些欄位、查詢邏輯——AI 直接沿用同樣的查法，不必重新摸索。

2

web.config （開權限給我們）

裡面是 Portal ↔ BPM SQL 的連線設定（伺服器／資料庫名等）。讓我們知道連到哪一台、哪個庫。

3

User DB Schema （表結構）

User Table 的欄位定義（欄位名／型別／代碼對照）——AI 才知道有哪些欄位可查、怎麼組查詢。（密碼欄已加密，無洩漏疑慮）

4

唯讀連線帳號 （建議新增）

一組只能 SELECT、只開 User Table（或專用 View）的帳號，讓 AI 中介程式安全連線。只讀不寫，不可能改到通訊錄。

待釐清

❓ 一項要順便問清楚

🔎

哪些欄位來自 AD、哪些來自 BPM？

目前確定分機由 AD 設定同步進來；但姓名／部門／工號／信箱／職稱／主管是來自 AD 還是 BPM（或 HR）尚未確認。這會影響「資料多久更新一次、以哪邊為準」，請 Scott 一併說明。

安全前提

🔒 我們怎麼確保安全

① 唯讀帳號：只能 SELECT，不能改任何資料。 ② 只開 User Table／專用 View，不碰其他表。 ③ AI 產生的查詢限定在這張表、且經中介程式把關。 ④ 通訊錄屬內部資料，走內網／私有 AI，不外送公開服務。

📅 本週進度

與 MIS 討論後釐清：通訊錄資料源＝BPM 的 SQL（User Table），由 AD 每天 2 次同步、Portal 的 PhoneSearch 讀取顯示，連線設定在 web.config。方向確定：AI 用唯讀帳號直接接 BPM SQL 做自然語言查詢。卡點：等 Scott 提供 4 項介接資訊／權限。
