nOCR × LimeIME 手寫辨識架構分析

從觸控事件到辨識結果 — Batch vs Incremental 完整技術報告

一、背景與系統總覽

問題：手寫辨識要怎麼運作？

我們的產品有兩個獨立的 App 需要協作：

  • LimeIME (LimeStudio) — 輸入法 App，提供手寫畫布讓使用者書寫，但自己不會辨識
  • nOCR — 辨識引擎 Service App，內嵌 MyScript iink SDK，負責將筆跡轉為文字

因為是兩個不同程序 (Process)，無法直接共用記憶體，所以透過 Android AIDL (Binder IPC) 進行跨程序通訊。

兩種辨識模式

我們實作了兩種辨識策略，各有優劣：

Batch 批次辨識Incremental 增量辨識
概念寫完全部 → 一次送出辨識邊寫邊辨識 → 即時顯示結果
體驗像寫完一句話再翻譯像語音即時翻譯，邊說邊出字
優勢實作簡單、辨識精確反應快、互動性高

以下將詳細說明兩種模式的架構差異。

三、AIDL 通訊機制 — oneway vs 同步

為什麼要用 AIDL？

LimeIME 和 nOCR 是兩個不同的程序 (Process)，Android 不允許直接呼叫對方的函式。
AIDL (Android Interface Definition Language) 是 Android 提供的跨程序呼叫機制，讓兩個 App 可以像呼叫本地函式一樣互相通訊。

同步呼叫（預設模式）

又稱 twoway（雙向），資料走「去程＋回程」：

Batch 用這種方式：呼叫 strokeTextCandidates() 後等辨識完才拿到結果。

oneway 呼叫（非阻塞模式）

資料只走「去程」，火箭發射後不回頭：

Incremental 用這種方式：每個觸控事件以 oneway feedPointerEvents() 發送，UI 完全不卡頓。結果透過 IOcrCallback 非同步推回。

為什麼 Incremental 必須用 oneway？

因為手指在螢幕上移動時，Android 系統每秒產生數十到上百個觸控事件。如果每個事件都用同步呼叫（等 nOCR 回應才繼續），UI 會嚴重卡頓、畫線斷斷續續。
oneway 讓事件「射後不理 (fire-and-forget)」，LimeIME 可以持續流暢地畫線和發送事件。

四、Batch 批次辨識流程

核心特徵：Editor 用完即丟

Batch 模式的關鍵是 MyScript SDK 的 pointerEvents(events, false)：

  • 第二個參數 false = 取代模式，每次呼叫會清除 Editor 裡所有舊筆跡
  • 既然每次都會清空，那 Editor 就沒必要保留，用完直接銷毀釋放資源
  • 背後的 ContentPackage(.iink)、Renderer 等資源也一併釋放，避免記憶體洩漏
  1. 使用者在 HandwritingView 上書寫，手指/筆離開 (ACTION_UP) 後觸發 onStrokeCollected()
  2. LIMEService 啟動 idle timer，等待使用者停止書寫（可設定延遲時間）
  3. Timer 到期，將所有累積筆跡序列化為 JSON 字串，透過 serializeStrokes()
  4. 同步 AIDL 呼叫 mOcrService.strokeTextCandidates(json)，阻塞等待結果
  5. nOCR 的 CommonMethods 建立新 Editor，以 pointerEvents(events, false) 一次性載入全部筆跡，辨識後 export 結果
  6. 銷毀 Editor — closeMyScriptEditor()
  7. 結果回傳 LimeIME，透過 handleHandwritingResult() 顯示於 CandidateView

五、Incremental 增量辨識流程

核心特徵：Editor 持久化存活

Incremental 模式使用 pointerEvents(events, true)：

  • 第二個參數 true = 累加模式，新筆跡追加到 Editor 現有筆跡上面，不清除
  • 因此 Editor 必須持續存活，如果銷毀就等於丟掉之前所有累加的筆跡
  • 用 IncrementalSession 包裝 Editor，用 SessionManager 管理其生命週期
  1. 使用者切換至增量模式，switchToIncrementalMode() 呼叫 beginIncrementalSession()
  2. nOCR 建立 IncrementalSession，持久化 Editor 不銷毀，回傳 sessionId
  3. 每個觸控事件 (DOWN/MOVE/UP) 透過 onRawTouchEvent() → onStrokeEventIncremental()
  4. 將 MotionEvent 轉換為 InkPointerEvent[]（含歷史事件），以 oneway AIDL 非阻塞呼叫 feedPointerEvents()
  5. nOCR 在背景以 pointerEvents(events, true) 累加式寫入筆跡
  6. 偵測到 UP 事件時自動辨識並透過 IOcrCallback.onIncrementalResult() 主動推送結果
  7. LIMEService 收到 callback，在主執行緒透過 handleIncrementalResult() 更新候選欄

六、Incremental 資料管線詳解

一筆觸控的完整旅程

以下說明使用者手指按下到辨識結果出現，資料經過的每一個 function：

① HandwritingView.onTouchEvent()

Android 系統自動呼叫 — 手指觸碰畫布時，系統傳入 MotionEvent，包含座標、壓力、時間戳。

重要概念：Android 會將短時間內多個觸控點打包成一個 MotionEvent（歷史事件），例如手指移動了 4 個位置，只收到 1 個 MOVE 事件，但裡面有 3 個歷史點 + 1 個當前點。

② onRawTouchEvent(MotionEvent)

轉接層 — HandwritingView 透過 listener 介面把原始觸控事件傳遞給 LIMEService，實現 UI 與業務邏輯的關注點分離。

③ onStrokeEventIncremental(MotionEvent) — 核心轉換

這是最關鍵的轉換步驟，做三件事：

步驟說明
動作對應ACTION_DOWN → InkPointerEvent.ACTION_DOWN
ACTION_MOVE → ACTION_MOVE
ACTION_UP → ACTION_UP
展開歷史事件將 MotionEvent 中的歷史點全部展開為獨立的 InkPointerEvent，確保辨識精度
時間校正將 uptimeMillis（系統啟動後）轉換為 currentTimeMillis（epoch 時間）

為什麼不能直接傳 MotionEvent？因為它是 Android UI 框架物件，無法通過 AIDL 跨程序傳輸。需轉成自定義的 InkPointerEvent（實作了 Parcelable）。

④ feedPointerEvents() — oneway AIDL 跨程序

透過 oneway AIDL 將 InkPointerEvent[] 送到 nOCR 程序。因為是 oneway，LimeIME 立即返回，繼續處理下一個觸控事件。

⑤ editor.pointerEvents(events, true) — MyScript 引擎

nOCR 的 IncrementalSession 將事件餵進 MyScript Editor（true = 累加模式）。偵測到 ACTION_UP 時，呼叫 editor.export_() 取得辨識結果，再透過 IOcrCallback.onIncrementalResult() 推回 LimeIME。

七、Session 生命週期詳解

為什麼需要 Session？

Batch 模式下，每次辨識都是「建立 Editor → 載入所有筆跡 → 辨識 → 銷毀 Editor」的一次性操作，沒有任何狀態保留。
但 Incremental 模式需要 Editor 持續存活，因為筆跡是一筆一筆逐步累加的，每次 feedPointerEvents() 都在同一個 Editor 上追加新筆跡。因此需要一個 Session 來管理這個 Editor 的生命週期。

Batch Editor 生命週期

✕ 沒有 Session 概念
✕ Editor 無法累加筆跡
✕ 每次都是從零開始

Incremental Session 生命週期

✔ Editor 持續存活，可累加筆跡
✔ 用 sessionId 識別各 Session
✔ 自動管理生命週期

Session 摧毀與重建機制

  • 清除畫布 (Eraser)：呼叫 clearIncrementalSession() → 結束當前 Session，但保持在增量模式。下次用戶開始書寫時自動建立新 Session。
  • 切換回 Batch：呼叫 switchToBatchMode() → endIncrementalSession() 完全關閉 Session + 清除狀態。
  • 離開輸入法：onFinishInput() → exitHandwritingMode() 自動清理。
  • nOCR 程序崩潰：DeathRecipient.binderDied() 觸發重新 bindOcrService()，下次書寫時重新建 Session。
