01E-Ink 物理特性基礎

在討論 Wait 模式之前，必須先理解 E-Ink 墨水粒子的物理翻轉特性 — 這是一切問題的根源。

🧪 LCD/OLED vs E-Ink 的物理差異

LCD / OLEDE-Ink (EPD)
像素切換電壓改變即瞬間切換 (~1ms)墨水膠囊中帶電粒子需物理位移 (~120-250ms)
驅動方式設定電壓後直接顯示需施加特定「波形」(一系列正負電壓脈衝序列)
被中斷後果無殘影，直接顯示新顏色粒子停在中間，留下「殘影」(Ghosting)
刷新率60~144 Hz~7-12 Hz (DU/A2 波形)

🔒
為什麼拖曳變得這麼慢？
每秒只能完成約 6~7 次渲染迴圈（1000ms ÷ 150ms ≈ 6.6 FPS）。正常 Android 是 60 FPS。
這意味著你的手指在螢幕上做了 60 個動作，但畫面只回應了 6 個，所以感覺極度「拖泥帶水」。
Wait = 1 效能瓶頸視覺化 — 1 秒內的時間分配

04兩模式直觀對比

核心的矛盾：要畫面乾淨就要卡住 UI，要 UI 貼手就會產生殘影。

🔒 Wait = 1 (阻塞模式)

  • DRM 驅動 Block 整個渲染管線
  • 每張 Frame 需等待 ~120-150ms
  • 實際 FPS：≈ 6-8 FPS
  • 殘影：完全無殘影
  • 操作手感：極度卡頓不貼手
  • 觸控事件大量堆積在 Queue

⚡ Wait = 0 (非同步模式)

  • DRM 驅動 立即返回不等待
  • 畫面送出速度：60 FPS
  • 實際 FPS：60 FPS
  • 殘影：嚴重波形碰撞殘影
  • 操作手感：極度流暢貼手
  • E-Ink 面板嚴重過載

💡 核心矛盾

E-Ink 的物理翻轉速度（120ms）與 Android 的渲染速度（16ms/幀）之間有 7.5 倍的差距。
Wait = 1 用「犧牲 UI 反應速度」來配合硬體；Wait = 0 用「犧牲畫面品質」來保持 UI 貼手。
兩種都是次優解。業界的頂尖方案是打破這個二元對立。

05業界頂尖方案 — 與我們的差距 (The Gap)

別家用一樣的 EPD 面板卻能做到「又貼手又乾淨」，他們到底多做了什麼？這裡的「Gap」指的就是我們與業界頂尖產品之間在底層驅動與硬體控制晶片上的技術差距。

📖
名詞解釋
The Gap（差距）：指我們目前的實作方式（簡單的 Wait / No-Wait 二擇一）與業界頂尖 E-Ink 產品（如 BOOX、reMarkable）之間，在底層驅動與硬體控制器技術上的落差。他們能同時兼顧「操作流暢」與「無殘影」，而我們目前只能二選一。

TCON（Timing Controller，時序控制器）：E-Ink 面板上的核心控制晶片，全名為 Timing Controller。它負責接收主機板送來的畫面資料，並轉換成驅動電子墨水粒子翻轉所需的精確電壓波形序列。高階 TCON（如 E-Ink T1000 或廠商自研 FPGA）內建大容量記憶體與波形碰撞處理引擎，能在硬體層級智慧處理快速連續畫面更新，避免殘影。低階 TCON 則只是被動接收指令，無法自行處理碰撞問題。
理想架構：非同步佇列 + 智慧合併 + TCON 波形碰撞處理
