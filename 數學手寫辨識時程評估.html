一、執行摘要

結論：技術完全可行。MyScript iink SDK 原生支援 Math Part，提供 15 種 Export 格式（含 LaTeX、MathML、PNG、JIIX 等），其中 Math Part 支援 5 種。

經分析 4 種實作方案，建議採用 Native PNG 渲染 + LaTeX 備用 (方案 D)，預估 6-8 天。

為何 JIIX 自繪品質不如 MyScript 原生 PNG？

JIIX 的 bounding-box 是 MyScript 用自己的字型和排版引擎算出來的。我們用 Paint.drawText() 去畫：

  • 字型不同 → 大小微偏
  • baseline 不同 → 上下錯位
  • 根號 √ 的弧線、括號 () 等無法用 drawText 完美還原
  • 分數線、上標/下標的精確對齊很難做到 MyScript 的品質

→ 自己畫的效果不如 MyScript 原生 PNG

八、MyScript 引擎內部原理

為什麼 MyScript Demo App 可以做到「擦掉部分符號、重新計算」的魔法？

這歸功於 MyScript 的 Interactive Ink（互動式墨水）技術和內部的 JIIX 樹狀模型：

Step 1：空間座標保留 — Tree Model

手寫 1 + 2 = 時，引擎在記憶體建立一棵「樹」：

節點1：筆跡 (辨識為 "1")  → 座標 (x:10, y:10)
節點2：筆跡 (辨識為 "+")  → 座標 (x:30, y:10)
節點3：筆跡 (辨識為 "2")  → 座標 (x:50, y:10)
節點4：筆跡 (辨識為 "=")  → 座標 (x:70, y:10)

Step 2：Convert — 偷天換日

editor.convert(DIGITAL_EDIT) 並不是截圖。而是：

  • 刪除原本的「手寫筆跡」節點
  • 替換成「數位字元」節點
  • 保留原始座標，在該座標上使用 STIX 字型渲染印刷體

在引擎眼裡，畫面上不是圖片，是一群「帶有座標和語意的數位物件」。

Step 3：橡皮擦 — 碰撞偵測

橡皮擦滑過 2 的區域 → 引擎計算碰撞 → 從樹中移除對應節點 → 重新排版。

Step 4：Live Solver — 即時重算

擦掉 2，手寫 5 → 引擎偵測新筆跡 → 觸發 Reflow → Solver 重算 1+5=6 → 自動更新結果。

為何 nNoteAS 目前做不到？

  1. 切斷物件連結：nNoteAS 把筆跡丟給 nOCR「黑箱」處理
  2. nOCR 完成排版：引擎內部建好完整的互動式樹狀結構
  3. 截圖成 PNG：由於架構設計（避免引擎佔用 nNoteAS 記憶體），nOCR 最終只是把樹「截圖成一張靜態 PNG」 → 拍遺照 (Rasterize)
  4. 貼回 nNoteAS：對畫布來說這只是一塊 Bitmap，無法知道哪裡是 1、哪裡是 3

→ 橡皮擦只能擦像素，無法通知引擎「使用者擦掉了 2，請重算 1+5」

💡 妥協方案：雙擊還原。使用者雙擊公式圖片 → 圖片消失 → 原始筆跡恢復 → 可重新編輯或重新辨識。以最小改動實現「可逆辨識」。
