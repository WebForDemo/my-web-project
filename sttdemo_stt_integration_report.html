sttDemo STT 串接實作報告

說明這次 sttDemo 如何把 App UI、Java Controller、JNI、Native Whisper / sean_stt runtime 串起來
專案：sttDemo
核心引擎：sean_stt / whisper.cpp
平台：Android + JNI + Native .so
報告時間：2026-05-08
1 整體目標

▎這次要做什麼

把 RD 已經能部署到設備上的 sean_stt / whisper.cpp 能力接進 Android App，讓 sttDemo 同時支援：

  • 即時 Mic STT
  • 音訊檔 STT（WAV / MP3 / 其他可解碼格式）
  • Debug 對照：同一段 Mic 錄音另存 WAV，再走檔案辨識比對

▎目前實作策略

不是自己重寫 STT 引擎，而是保留既有 native 模型與 runtime，App 端主要負責：

  • UI 互動與權限
  • 錄音 / 檔案解碼
  • PCM / float 音訊資料搬運
  • JNI 橋接與結果顯示

4 啟動與模型載入流程
Step 1. Activity 建立 Controller

SttDemoActivity.onCreate() 建立 RealtimeSttController，並呼叫 initialize()。

Step 2. AssetUtils 複製模型與 encoder.dla

把 assets/models/ggml-base-q8_0.bin 與 ggml-base-q8_0-encoder.dla 複製到 /data/user/0/.../files/。

Step 3. NativeWhisperLoader 抽出 .so 並依序載入

先把 libomp.so、libggml*.so、libneuron_runtime.so.8、libwhisper.so 從 assets 複製到私有目錄，再用 System.load() 載入，最後才 System.loadLibrary("stt_jni")。

Step 4. JNI 建立 whisper context

nativeCreate() 呼叫 whisper_init_from_file_with_params()，成功後回傳 native handle 給 Java。

Step 5. UI 顯示 Model ready

Controller 透過 callback 通知 Activity，啟用 `Start Mic STT` 與 `Pick Audio File` 按鈕。

5 Realtime Mic 流程
1. 使用者按下 Start

Activity 先處理 RECORD_AUDIO 權限，成功後呼叫 sttController.start()。

→
2. AudioRecord 開始收音

設定為 16kHz / Mono / PCM16，資料來源目前使用 VOICE_RECOGNITION。

→
3. PCM16 轉 float

每次讀到的 short[] 轉成 float，累積到本輪 utterance buffer。

→
4. 每隔固定步進做一次完整推論

目前不是只取最後一小窗，而是把「本句目前已收錄的全部音訊」做一次完整 transcribeFile()，藉此提高字詞穩定度。

5. 產生 Live Result

最新一版完整假設文字顯示在 `Live Result`，讓使用者看到當下辨識內容。

→
6. 取穩定前綴

把前一次假設與本次假設做最長共同前綴比較，只有穩定下來的前半段才更新到 `Transcript`。

→
7. 偵測停頓後 commit 整句

若靜音超過約 1.2s，把目前這句最終結果追加進全文 Transcript，並清空本輪 utterance 狀態。

→
8. Stop 時 final flush

按下 Stop 會先做最後一次推論，再把最後一句 commit，最後才進入 debug WAV 對照流程。

這裡有一個很重要的設計取捨

雖然 native 端仍保留 transcribeStream() 與 stream prompt token 狀態，但目前 RealtimeSttController 的 realtime 主路徑是用 transcribeFile() 去重跑「本句目前整段音訊」。原因很直接：實測下，這條路比直接吃 streaming partial 結果穩定，較接近下方 Recorded WAV Result 的品質。

6 Audio File 流程
1. 使用者按下 Pick Audio File

Activity 透過 OpenDocument 讓使用者選取 WAV / MP3 / 其他音訊檔。

→
2. AudioFileDecoder 解碼

若是 WAV，直接走 WavFileDecoder；若是 MP3 / 壓縮格式，走 MediaExtractor + MediaCodec 解碼。

→
3. 統一轉成 mono 16k float[]

所有輸入最後都會被整理成 Whisper 可接受的 float[] 音訊。

→
4. NativeWhisper.transcribeFile()

走 non-streaming 參數做完整推論，結果直接回傳給 Activity 顯示到 Transcript。

7 JNI 與 Native 邊界

JNI NativeWhisper.java 的角色

  • 保存 native handle
  • 對 Java 層提供簡單 API：transcribeFile()、transcribeStream()、resetStreamState()
  • 避免 Activity 或 Controller 直接碰 native 細節

JNI stt_jni.cpp 的角色

  • 建立 / 釋放 whisper context
  • 填入 whisper_full_params
  • 把多 segment text 拼成單一字串
  • 維護 stream prompt tokens，讓 streaming mode 有上下文
  • 做 UTF-8 清理，避免 NewStringUTF() 因不完整中文字節點直接 crash

目前重要 native 參數

params.translate = false; params.no_context = !streaming_mode; params.single_segment = streaming_mode; params.max_tokens = 0; params.suppress_nst = true; params.temperature = 0.0f; params.temperature_inc = 0.2f; params.no_timestamps = true;

也就是說：檔案模式傾向完整轉寫；streaming mode 才保留 prompt tokens 與 single-segment 行為。

8 UI 顯示邏輯
UI 區塊 資料來源 用途
Status onStatusChanged() 顯示 `Loading model`、`Listening...`、`Comparing recorded WAV...`、`Stopped` 等狀態。
Live Result onLiveTextChanged() 顯示當下最新一輪 realtime 假設，允許變動。
Transcript onTranscriptCommitted() 顯示穩定累積下來的全文，不應只剩最後一句 partial。
Recorded WAV Result onRecordedWavResult() 顯示 Mic 停止後另存 WAV 再跑一次檔案模式的結果，用來和 realtime 做直接對照。
9 Debug 對照機制
Mic 錄音同時寫入 ByteArrayOutputStream

這份原始 PCM 不只用來 realtime 推論，也用來做停止後的驗證。

→
Stop 後寫成 WAV

WavFileWriter 會把 PCM16 包成標準 WAV，輸出到 /files/debug/last_mic_recording.wav。

→
再走一次 File STT

用 WavFileDecoder 讀回來，丟給 NativeWhisper.transcribeFile() 跑完整辨識。

→
顯示 Recorded WAV Result

這是目前最重要的品質對照基準。若它準而 realtime 不準，問題就在 realtime pipeline，而不是模型本身。

11 目前狀態與限制

已經驗證

  • 模型與 native runtime 已成功接進 App。
  • WAV / MP3 檔案辨識可運作。
  • Mic 錄音可被另存成 WAV，並用來做品質對照。
  • Transcript 已改成全文累積，不只是最後一段 partial。

仍待收斂

  • Realtime Mic 效果仍明顯弱於下方 Recorded WAV Result。
  • 表示問題仍集中在「邊錄邊推論」節奏與策略，不在模型本身。
  • 目前走的是比較保守的「整句增量重跑」方案，準確度與延遲還需要繼續取平衡。
12 老闆講稿

▎1 分鐘版

這個 app 的核心是把既有的 native Whisper 能力，透過 JNI 串接到 Android App 裡。上層是 UI，負責開始錄音、選音訊檔、以及顯示 Live Result、Transcript 跟 Recorded WAV Result。中間層是 Java controller，負責錄音、buffer、切句、以及 Transcript 累積。再往下是 JNI，把 Java 的音訊資料送進 native。最底下則是 sean_stt / whisper runtime，實際執行辨識。

目前檔案辨識路徑已經穩定，realtime mic 也已經串通，但品質還在收斂中。從目前對照結果來看，模型本身是可用的，接下來主要優化的是邊錄邊推論的策略，而不是重做 STT engine。

▎3 分鐘版

這套系統可以分成五層。第一層是 Android UI，使用者透過按鈕開始錄音或選音訊檔，畫面上會看到即時文字、全文 Transcript，以及停止錄音後的 WAV 對照結果。

第二層是 Java control layer。這一層是整個 app 的控制核心，負責 AudioRecord 收音、把 PCM16 轉成 float、管理本句 buffer、做停頓判斷，以及決定什麼時候觸發推論。也就是說，realtime 行為的主要調度不在 JNI，而是在 Java 這一層。

第三層是 JNI bridge。這一層把 Java 的 float[] 音訊送進 native，並把 native 回來的字串安全地交還給 Java。這裡也處理過 UTF-8 安全問題，避免中文字尾端不完整造成 crash。

第四層是 native runtime，也就是 sean_stt / whisper.cpp。這一層真正跑 whisper_full() 推論，並且在 stream 模式下保留 prompt token state，讓前後 chunk 之間有上下文。

第五層是裝置 runtime 與硬體。目前實際狀態是 APU encoder 加上 CPU decoder fallback，不是完全所有步驟都跑在 NPU 上。這也是為什麼現在完整檔案辨識的品質比 realtime mic 更穩定。

如果看目前成果，音訊檔辨識已經證明模型可用，realtime mic 也已經串接完成。下一步重點就是持續優化 realtime 推論節奏與切句策略，讓它更接近 Recorded WAV Result 的品質。

▎逐層指圖講稿

  1. 先看最上面 UI Layer，這層就是使用者直接看到的 app 畫面，包括 Live Result、Transcript、以及 Recorded WAV Result。
  2. 再看第二層 Java Control Layer，這裡是整個 app 的控制核心，包含 AudioRecord、Utterance Buffer、Audio File Decoder、Debug WAV Writer，realtime 的 buffer、停頓判斷、全文累積邏輯都在這裡。
  3. 第三層 JNI Bridge Layer 比較像橋樑，不負責 UI，也不直接負責使用者流程，它只是把 Java 的 float[] 資料送進 native，然後把 String result 帶回來。
  4. 第四層 Native Runtime Layer 就是真正的 STT 引擎，包括 libwhisper / sean_stt、whisper_full()、UTF-8 sanitize，以及 stream mode 的 prompt token state。
  5. 最底下是裝置硬體層，目前比較準確的說法是 APU Encoder 加 CPU Decoder Fallback，也就是部分硬體加速、部分 CPU 執行。
  6. 如果看資料流，麥克風路徑是 AudioRecord 收音後進入 Java buffer，再由 controller 定時送進 JNI 和 native，最後文字結果回到 Java 更新畫面。
  7. 如果看檔案路徑，就是選取 WAV 或 MP3 後先解碼，再送 native 做完整辨識，直接更新 Transcript。
  8. 如果看 debug 路徑，就是 mic 錄到的 PCM 另外寫成 WAV，再走一次 file transcription，用來判斷問題是在模型還是在 realtime pipeline。
