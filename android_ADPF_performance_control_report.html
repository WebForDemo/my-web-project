3 Android 效能控制方案總覽
序號方案名稱核心技術 / API控制層級「開書」場景可能應用
1ADPF (動態效能架構)PerformanceHintManager微秒級 (執行緒)點擊瞬間引導 CPU 垂直升頻，縮短解析時間。
2Game Mode APIGameManager應用級 (全局)設定成 PERFORMANCE 模式，讓系統放寬溫控與排程門檻。
3Thread PriorityProcess.setThreadPriority核心調度層避免背景工作搶走「開書」執行緒 CPU 時間。
4Power ManagersetSustainedPerformanceMode硬體穩定層避免長時間解析大檔時突然斷崖降速。
5WakeLockPowerManager.WakeLock電源狀態層防止 CPU 在高負載運算中進入省電休眠。

▎ADPF

Android 13+ 提供的雙向溝通機制。App 告訴系統預期工作時間，系統則回饋熱狀態與調度協助，是目前唯一能主動引導 CPU 頻率爬升的現代 API。

▎Game Mode API

雖然名稱帶有 Game，但本質上是效能偏好查詢機制。系統可依 PERFORMANCE / STANDARD / BATTERY 切換不同 PowerHAL 配置。

▎Thread Priority

底層 Linux 調度優先權。它不直接升頻，但能在 CPU 資源不足時優先處理關鍵工作執行緒。

▎Sustained Performance / WakeLock

前者維持較穩定但不極端的頻率曲線，後者確保背景解析時 CPU 不進入休眠；兩者都偏系統穩定性而非瞬時爆發。

▎測試目的

驗證 use ADPF 開啟與關閉時，ADPF Sample 在相同執行時間下的熱狀態、FPS 目標與平均耗電差異。

PDF 內對 workload 的簡化定義是：裝置在一段時間內持續要處理的工作量，例如持續運算、渲染畫面、物理模擬、吃 CPU/GPU 的任務。

▎測試方法

  • 測試 App：ADPF Sample
  • 測試方式：比較 use ADPF = ON 與 OFF
  • 操作方式：每次切換模式後按 Reset 重新統計
  • 主要觀察值：FPS target、Thermal State、平均耗電

▎測試步驟

Step 1

開啟 ADPF Sample。

Step 2

確認兩輪測試條件一致，使用相同裝置、相同執行場景與相同觀察時間。

Step 3

第一輪勾選 use ADPF，按下 Reset，開始計時 3 分鐘。

Step 4

記錄 FPS target、Thermal State、平均耗電值。

Step 5

第二輪取消勾選 use ADPF，再按一次 Reset，同樣觀察 3 分鐘。

▎重點結論

  • ADPF 可有效延後裝置進入較高熱狀態，從 2:44 延後到 6:22 才出現相同行為。
  • ADPF 可延長高效能維持時間，在相同持續工作量下維持 60 FPS 更久。
  • 平均耗電從 228094 降到 220759，約改善 3.2%。

▎適用情境判讀

PDF 明確指出：ADPF 不太適合拿來當單次「開書」動作的主要方案。它更適合遊戲、長時間渲染、持續運算、長時間高 CPU/GPU 任務等持續負載場景。

▎結論與適用情境

  • Game Mode Sample 可正確接收系統設定的遊戲模式。
  • App 會依不同模式調整 FPS 與解析度策略。
  • 適合遊戲、即時渲染、模擬類 App、長時間持續運算的互動式內容。
  • 不適合一次性短操作，例如單次開頁、單次載入、短暫查詢等流程。

7 Game Mode 與 ADPF 適用情境比較總結
比較項目Game Mode (遊戲模式)ADPF (動態效能架構)
核心適用產品 / 場景 遊戲、即時渲染、模擬類 App、需要讓使用者切換「效能 vs 省電」偏好的場景。 遊戲、長時間渲染、需要穩定 CPU/GPU 輸出的持續性高負載任務。
主要控制 / 調節目標 宏觀策略：降解析度、鎖 FPS、減少特效、降低背景更新。 微觀調度：回報實際運算時間，讓系統動態調節頻率以維持長時間穩定。
對於「單次開書」的適用性 ❌ 不適合 ❌ 不太適合當主要方案
不適合「開書」的核心原因 開書是「一次性、單一步驟」的短操作。Game Mode 是給「持續運行的策略」用的。 開書是「短時間的爆發運算」。ADPF 擅長處理「持續性的高工作量」，對突發任務反應不及。
