二、Special Refresh 四個參數

PowerManagerService 從 Broadcast Intent 中提取以下四個 extra，傳入 Surface.updateAuxRefreshModeArg(NTX_MODE_SPECIAL, value, count, time, region)

① Value
intent.getIntExtra("updatemode", 0)
Waveform 模式 — 指定這次 Special 刷新要使用的 waveform。 常見值：
• GC16_NO_MERGE — 高品質灰階 16 階（全刷新清殘影）
• GLR16 — 灰階低閃爍 16 階（閱讀翻頁）
• GLR_FULL — GLR16 + 全螢幕 + Wait
✓ 目前有使用
② Count
intent.getIntExtra("apply_count", 0)
套用次數 — 指定此 Special 模式要持續套用幾次畫面更新事件。
• 0 = 預設行為（用一次就清除）
• N = 持續 N 次更新後才清除

設計目的：解決「彈窗加速時 Special 被一次 update 消耗掉」的問題 (#5032)。
✗ Sender 端未使用（預設 0）
③ Time
intent.getIntExtra("drop_time", 0)
Update Drop 時間 (ms) — 搭配 Drop Mode 使用。在此期間 Special 模式不會被清除。
• 0 = 不啟用 drop
• 500+ = Drop 持續毫秒數（翻頁場景常用 500ms 以上）

用途：過濾翻頁動畫中間幀，只讓最終結果走 Special waveform
✓ 透過 broadcastEinkModeDropTime() 使用
④ Region
intent.getIntExtra("region", 0)
刷新區域標識 — 指定此次 Special 刷新是全螢幕還是局部區域。
• 0 = 預設（由系統根據 dirty region 判斷）
• 其他值 = 強制指定區域類型

這個參數讓 GLR16 可以繞過「非 Full Region 不走 GLR16」的底層限制。
✗ Sender 端未使用（預設 0）

六、HWComposer 的清除邏輯

在每次 presentAndGetReleaseFences() 呼叫中（螢幕 present 一幀畫面時）：

// HWComposer.cpp — presentAndGetReleaseFences() // 1. 先讀取當前 Quick 值 uint32_t mQuickMode = displayData.auxRefreshMode->mQuickMode; // 2. 送到硬體 auto err = hwcDisplay->setAuxRefreshMode(displayData.auxRefreshMode); // 3. Quick → 無條件清除 if (mQuickMode) displayData.auxRefreshMode->mQuickMode = 0; // 4. Special → 只在 DropFrame 未啟用時才清除 if (displayData.auxRefreshMode->mDropMode != DROP_FRAME_ENABLE) { displayData.auxRefreshMode->mSpecialMode = 0; displayData.auxRefreshMode->mCount = 0; displayData.auxRefreshMode->mTime = 0; displayData.auxRefreshMode->mRegion = 0; }
💡 關鍵差異就在這裡：
• Quick — if (mQuickMode) mQuickMode = 0 → 無條件清除，一幀後就消失
• Special — if (mDropMode != DROP_FRAME_ENABLE) → DropFrame 啟用期間不清除，讓 Special waveform 跨多幀持續生效
• Global — 完全不在此處清除，只有收到 RESET broadcast 時在 Java 層手動清

// Activity.java static final int EINK_GLR_FULL = EINK_GLR16_MODE // 0x40 — GLR16 waveform | EINK_FULLSCREEN_UPDATE // 0x10000000 — 強制全螢幕（Regal 需要） | EINK_WAIT_UPDATE; // 0x02000000 — 等刷完才返回（防撕裂）
💡 為什麼 Regal 需要全螢幕？
Regal 演算法需要比對前後兩幀的完整畫面差異來計算殘影補償向量。如果只做局部更新，TCON 無法取得完整差異資訊，Regal 就會退化成普通 GL16（有殘影）。
所以 EINK_GLR_FULL 一定要帶 EINK_FULLSCREEN_UPDATE，強制 dirty region = 整個螢幕。

為什麼 Dirty Region、TCON、波形碰撞這些概念在 E-Ink 上特別重要，在 LCD 上卻幾乎不需要關心？

9.1 刷新速度 — 根本差異

所有 E-Ink 特有問題的根源都在於：E-Ink 單次刷新比 LCD 慢 30～40 倍。

💡 比喻：
LCD 像快乾白板筆 — 寫了馬上乾，可以立刻擦掉重寫

• 工作量：動態高

9.4 波形碰撞（Waveform Collision）— E-Ink 特有

當前一次刷新還沒完成，下一次刷新就開始，兩組電場波形在同一區域疊加 → 殘影、閃爍、灰階錯誤。
