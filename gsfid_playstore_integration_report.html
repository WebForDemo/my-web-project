Play Store GSFID 串接報告

Android 16 MtkSettings 與 Vendor init/script 的整合設計、問題定位與最終修正
System Tree：android16_mtk_system_b0mp5_aiv
Vendor Tree：android16_mtk_vendor_v0mp5_aiv
核心重點：Settings 發 request，Vendor 高權限查 DB，結果再回推 UI
1 問題背景

▎原始需求

將 Android 12 的 EnablePlayStoreFragment / GSFID 顯示流程移植到 Android 16，並確保當裝置已經產生 GSFID 時，Play Store 頁面會顯示 GSF ID、Step 2 與註冊導引畫面。

▎遇到的真實問題

  • Android 16 上，原本的 provider 查詢 content://com.google.android.gsf.gservices 不可靠。
  • 實機驗證發現真正有值的是 /data/data/com.google.android.gms/databases/gservices.db 的 android_id。
  • 只有手動執行 script 時，GSFID 才會被抓出來並讓畫面切換；正常使用設備時，service 沒有成功以正確權限跑起來。

3 問題根因與修正
Before
  • Settings 端曾嘗試直接使用 vendor.debug.* property，造成權限錯誤。
  • 正常流程雖然有設到 request property，但 init service 沒有以正確權限上下文執行 script。
  • 結果是：手動 su 0 sh /vendor/etc/fetch_gsf_id.sh 有效，正常 UI 流程無效。
After
  • Settings 改成只使用 debug.gsf_fetch_requested / debug.gsf_id。
  • Vendor service 最終明確使用 seclabel u:r:su:s0，與手動成功路徑一致。
  • 正常進頁面時就能啟動抓取流程，抓到 GSFID 後自動切換畫面。

5 實際資料來源與指令

▎實機最終確認

Android 16 這台裝置上，GSFID 實際是從以下 DB 查到：

/data/data/com.google.android.gms/databases/gservices.db

實際查詢指令：

sqlite3 /data/data/com.google.android.gms/databases/gservices.db "select value from main where name='android_id';"

▎腳本成功後的輸出

  • Property：debug.gsf_id
  • Cache：/data/vendor/gsf/gsf_id
  • Broadcast：com.android.settings.action.GSF_ID_UPDATED
6 使用情境說明

▎正常使用設備

  • 使用者進入 Play Store 設定頁。
  • 若裝置當下已有 GSFID，UI 會顯示 GSF ID 畫面。
  • 若當下沒有，使用者去開 Play Store、登入或初始化後，再回來頁面，onResume() 會再觸發抓取流程。

▎手動測試 / CMD 測試

  • 送 request broadcast：adb shell am broadcast -a com.android.settings.action.REQUEST_GSF_ID_FETCH
  • 直接設 trigger property：adb shell setprop debug.gsf_fetch_requested test1
  • 直接執行 script：adb shell su 0 sh -x /vendor/etc/fetch_gsf_id.sh
7 結論

這次整合的核心不是單純把 Android 12 的 UI 邏輯搬過來，而是重新拆清楚權責：

  • Settings 只負責發 request、收結果、更新畫面。
  • Vendor init/script 才負責高權限讀取 SQLite DB。
  • 關鍵修正 是讓 init service 以 u:r:su:s0 執行，與手動成功路徑一致。

最終結果是：正常進入 Play Store 設定頁時，不需要手動跑 script，也能自動抓到 GSFID 並切換成正確畫面。
