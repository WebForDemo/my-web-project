01 · 問題背景

為什麼需要這套機制？

E-Ink 裝置的手寫出水行為若無法精確控制，將直接影響使用者體驗與產品品質。

現有問題

傳統手動管理的痛點

過去依賴 App 工程師在每個可能遮擋畫布的地方，手動呼叫 suspendHandWrite()。

  • 容易在新功能開發時遺漏呼叫點
  • 不同 App 實作品質參差不齊
  • Dialog、Fragment、PopupWindow 各需個別處理
  • 維護成本高，每次 UI 改版都需重新確認
  • 測試難以覆蓋所有邊緣情境
本 PoC 解法

自動化、防呆的底層機制

在 Java Framework 層建立統一的出水決策中樞，App 端完全不需要手動介入。

  • 非侵入式設計，不修改任何 App 程式碼
  • 三層防護自動偵測所有遮擋情境
  • 同一個邏輯服務所有 App
  • 新 App 開發無需特別注意出水問題
  • 提供 Hook 機制保留 App 端特殊需求的彈性

Toggle Fragment

🪟

情境 B — 跨 Window 懸浮視窗

Show PopupWindow

以 isFocusable=true 建立 PopupWindow 顯示在畫布中央。系統在後續的 touch 事件中附加 FLAG_WINDOW_IS_PARTIALLY_OBSCURED，由第二層 B 捕捉。

第二層 B：MotionEvent Flag
🚨

情境 C — 系統 Dialog 搶焦點

Show AlertDialog

AlertDialog 彈出時系統自動搶走 Window Focus，MockDrawView 的 hasWindowFocus() 立即返回 false，由第二層 A 捕捉，不依賴任何 touch 事件。

第二層 A：Window Focus
🖊️

情境 D — App 業務邏輯強制介入

Eraser Mode ☑

勾選橡皮擦模式後，Hook 回傳 true 強制關水，不論底層掃描結果為何。模擬 App 需要在特殊狀態下完全自主控制出水的場景。

06 · 已知盲點與對策

邊緣情境與處理方式

以下情境超出自動偵測範圍，需要 App 端配合或額外說明。

PopupWindow isFocusable=false

若 PopupWindow 設為不可取焦，將不觸發 Window Focus Lost；部分 Android 版本對同 UID 的視窗豁免 FLAG_WINDOW_IS_PARTIALLY_OBSCURED，導致第二層完全失效。

App 端必須透過第三層 BusinessLogicHook 手動強制關水。

固定工具列誤判

App 中常見的固定工具列（Toolbox）位置固定且始終疊在畫布上方，若不特別處理，第一層掃描會永遠判定遮擋導致全局關水。

在包裹 Toolbox 的 ViewGroup 加上 android:tag="ignore_obscure"，掃描時自動略過。

透明 ViewGroup 容器

第一層掃描遇到無背景的透明 ViewGroup 時，不會直接判定遮擋，而是繼續掃描其子 View，避免因容器節點產生誤判。

設計上已處理；只有 Leaf View 或有背景的 ViewGroup 才算實體遮擋。

系統懸浮視窗（System Alert Window）

部分系統級懸浮視窗（如無障礙服務覆蓋層）屬於不同 UID，行為與一般 App Window 不同，旗標保護機制可能不完整覆蓋。

此情境屬於系統層級問題，建議由 BSP 在 Framework 中另行處理。
