1 需求與現象

現象
App（ntx.note2）已設定全螢幕、隱藏 status bar；但使用者從螢幕上緣往下滑，狀態列仍會被「暫時」拉出來（transient status bar）。
需求
只針對這一個 App，把「上緣下滑叫出 status bar」這個手勢擋掉；其他 App、以及 navigation bar 不受影響。
⚠️ 純 App 端做不到：全螢幕 immersive 下「邊緣滑一下露出系統列」是系統層保證的行為，公開 API 只能選行為、無法完全禁止露出。要真正擋掉必須動 framework（本專案有 ROM 原始碼，可行）。

2 背景：全螢幕與 Transient Bars

全螢幕 / Immersive
App 呼叫 WindowInsetsController.hide(statusBars()) 把狀態列隱藏，畫面延伸到全螢幕。
Transient Bars
隱藏後，使用者從邊緣滑一下，系統會暫時顯示系統列（一會兒自動縮回），這是預設的可用性保證。
誰決定露出
由 WindowManager 的 DisplayPolicy 收到手勢後決定，不是 App 自己。所以要擋，得在這一層。

為什麼不改 App / 不改 ImmersiveModeConfirmation

  • BEHAVIOR_SHOW_TRANSIENT_BARS_BY_SWIPE 只能決定「滑動要不要顯示」的行為模式，sticky immersive 仍會短暫露出，沒有公開旗標可完全禁止。
  • ImmersiveModeConfirmation 管的是「滑動可呼出系統列」的提示泡泡，拿掉它只是不提示，露出本身照樣發生。
  • 結論：要擋「露出」本身，正確位置是 DisplayPolicy 收手勢、要不要 requestTransientBars 的那一步。

4 攔截點：在源頭直接擋掉

修改前
onSwipeFromTop() {
  requestTransientBars(mTopGestureHost, …);
}
// 任何 App 上緣下滑都露出 status bar
修改後
onSwipeFromTop() {
  if (focus 的包名 == "ntx.note2")
      return;   // 直接不做
  requestTransientBars(mTopGestureHost, …);
}

怎麼認出「是不是 ntx.note2」

  • DisplayPolicy 有欄位 private WindowState mFocusedWindow;（:280），隨焦點更新。
  • WindowState.getOwningPackage()（WindowState.java:1237）回傳 mAttrs.packageName = 建立該視窗的 App 包名。
  • 比對 mFocusedWindow.getOwningPackage() 是否等於 "ntx.note2"，只針對它 return。

5 具體修改（唯一一處）

frameworks/base/services/core/java/com/android/server/wm/DisplayPolicy.java · 方法 onSwipeFromTop()

                 @Override
                 public void onSwipeFromTop() {
                     synchronized (mLock) {
+                        // NTX: for this app, block the swipe-down-from-top gesture from
+                        // revealing the (transient) status bar. Only affects the status
+                        // bar (top edge); nav bar / other edges are unaffected.
+                        if (mFocusedWindow != null
+                                && "ntx.note2".equals(mFocusedWindow.getOwningPackage())) {
+                            return;
+                        }
                         requestTransientBars(mTopGestureHost,
                                 getControllableInsets(mTopGestureHost).top > 0);
                     }
                 }
只加 6 行判斷,不動任何既有邏輯;判斷不成立就走原路,對其他 App 零影響。

6 效果與範圍

情境行為說明
ntx.note2 上緣下滑已擋不 requestTransientBars → status bar 不露出
其他 App 上緣下滑照舊判斷不成立,走原本邏輯,正常露出
navigation bar（下緣上滑）不影響那是 onSwipeFromBottom,本次沒動
ntx.note2 的通知欄 / QuickSettings連帶擋掉全螢幕下「下滑叫通知欄/QS」是先靠這條露出 status bar,擋掉後同一手勢也進不去
此為先前確認的「全擋」取捨：只要 從上緣下滑,ntx.note2 一律沒反應。若日後想「擋 status bar 露出、但仍保留可拉通知欄/QS」,需另外改 SystemUI 的 shade 展開路徑開例外(不在此處)。

7 層級、編譯與驗證

屬於哪一層
  • Framework / WindowManagerService（system_server）,打包在 services.jar。
  • 不是 SystemUI、也不是 App 端。
  • 用「前景視窗包名」判斷,ntx.note2 本身不需修改。
編譯
  • 改 system 側(system_server / services.jar)。
  • 重編 system image 後燒入即可,不需動 kernel / vendor。

驗證方式

  • 進 ntx.note2 全螢幕畫面,從上緣下滑 → status bar 不該出現、通知欄/QS 也拉不出來。
  • 換其他 App 上緣下滑 → status bar 應正常露出(確認沒誤傷)。
  • 下緣上滑 → nav bar 應正常(確認只擋上緣)。
  • 需要時可加 log:在 onSwipeFromTop return 前印 getOwningPackage() 確認辨識正確。

8 總結

項目內容
目標只讓 ntx.note2 全螢幕時「上緣下滑」不露出 status bar
根因transient bar 露出由系統(DisplayPolicy)決定,App 端無公開 API 可禁止
做法在 DisplayPolicy.onSwipeFromTop() 依前景包名攔截,略過 requestTransientBars
影響檔案DisplayPolicy.java(唯一一處,+6 行)
範圍只 ntx.note2、只 status bar;nav bar 與其他 App 不受影響
連帶該 App 從上緣下滑的通知欄 / QS 也一併進不去(全擋)
最小侵入:單檔 6 行、走前景包名判斷,不改 App、不改 SystemUI、不動 kernel/vendor。
