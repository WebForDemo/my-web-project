4 決策結論
Recommendation
建議對外展示可採用，但正式客戶場合應避免使用中文長句作為核心 Demo。
系統在串流穩定性面已過關，因為漏字與重複資料皆未出現；現階段真正會拉低專業感的，是中文內容中的語意替換錯誤。若要給大老闆或客戶看，建議先以英文短句、英文長句與受控長度中文腳本進行展示。
展示可用性: 中高
中文長句風險: 高
英文展示成熟度: 高
Immediate Focus
優先修正中文詞替換 `語音`→`與因`、`家電`→`加電` 這類錯誤最影響理解與觀感。
展示腳本分級 Demo 場景先使用低風險英文樣本與短中文句，避免長中文段落。
下一輪驗證方向 增加中文專有名詞、口語停頓與連續長句測試，確認模型邊界。
5 測試方法與限制
Test Method
本次結果是「檔案回放辨識」基線測試，不是最終 Real Mic In 使用體驗。
測試流程為先將文字轉成 MP3，將 MP3 放置到設備端，再透過 `curl` 呼叫 STT 辨識介面。這種方法的價值在於可穩定比較不同版本模型，但因為沒有經過真實現場收音條件，所以結果通常會比實際使用情境更理想。

未覆蓋: Real Mic In 情境

Gap To Product
麥克風與聲學環境差異 真實收音會受到麥克風頻響、距離、方向性、環境噪音、回音與殘響影響。
前處理鏈路可能不同 AGC、AEC、NS、VAD、音量飽和與底噪抑制等處理，未必與檔案輸入路徑完全一致。
真人說話變數尚未驗證 語速、停頓、口音、咬字、情緒與臨場語調，都可能讓實際結果低於本次報告。
Development Recommendation
現階段建議不要把本次結果直接解讀為「設備端已完成 Real Mic In」。
依據設備端工程師的說明，目前程式碼裡雖然找得到 mic input 的 demo 方向，但原始實作是使用 `SDL2` 抓音，並不是 Android 平台可直接上線的收音方式。若要真正落地到設備端，仍需改成 Android `AudioRecord`，且會牽涉 source code 與 build rule 調整。
現況: 已能做檔案回放 STT 驗證
差距: 尚未等同 Android Real Mic In 成品
重點: 需要額外開發，不是直接開關啟用
短期建議 對內與對外簡報可先明確定位為「檔案回放基線測試」，不要宣稱已完成真實麥克風即時輸入。
中期建議 若目標是設備端直接收音，可評估把 mic capture 改為 Android `AudioRecord`，但需預留 source code 與 build 整合時間。
正式產品建議 較合理的產品化方向，是由 app 端用 `AudioRecord` 收音，再以 streaming 方式送到同設備內的本地 STT service，由 service 回傳即時分段結果。這比直接硬改 demo 更接近正式架構。
