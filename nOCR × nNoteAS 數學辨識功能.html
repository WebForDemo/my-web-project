專有名詞解釋

LaTeX — 文字版的簡譜

你手寫了 ½，辨識引擎給你的是一串純文字密碼：\frac{1}{2}

  • 全名：Lamport TeX
  • 定位：排版語言標準
  • 缺點：只是文字，普通人看不懂

KaTeX — 網頁版的翻譯機

舊作法 在 App 裡開一個隱藏的 WebView，把 LaTeX 字串丟給 KaTeX 翻譯成數學圖形。

  • 全名：Khan Academy TeX
  • 定位：JavaScript 渲染函式庫
  • 缺點：開網頁慢、耗效能、排版有時怪

階段一：觸發與收集筆跡 (nNoteAS 端)

使用者在畫布上框選手寫算式，並點擊「數學辨識」按鈕：

  1. HandwriterView 收集被選取的筆跡（List<Graphics> selectedStrokes），排除螢光筆
  2. 將筆跡的 x, y 座標打包為 ArrayList<StrokePoints>，用 Gson 序列化為 JSON
  3. 若筆跡數 ≥ 100 → 先寫入暫存檔（避免 AIDL Binder 傳輸上限）
  4. 透過 AIDL 呼叫 NtxNoteApplication.recognizeNooseSelectMathResult(json, enableSolver)
  5. 內部先呼叫 screenDirection() 設定螢幕方向，再呼叫 strokeMathDataWithSolver()，同步阻塞等待結果

階段二：MyScript 引擎魔法 (nOCR 端)

🔑 這是此次改版最核心的地方，所有的「排版魔法」都在這裡發生。

  1. 初始化 Math Editor

    呼叫 getMyScriptEditorMathType()，核心差異是 ContentPart("Math")（文字辨識用 "Text"）

  2. 套用排版主題 (Theme CSS)

    editor.setTheme(".math { font-family: STIX; } .math-variable { font-style: italic; }");

    告訴引擎：「用專業的 STIX 字型畫符號，且變數要斜體」。這是 Demo App 能產出精美公式的關鍵。

  3. 設定 Math Solver

    setMathSolverEnable(enableSolver) → 設定 math.solver.enable。啟用後 SDK 不僅辨識公式，還嘗試計算結果（例如 2+3= → 自動補上 5）

  4. 筆跡注入 + 排版轉換 Math 獨有

    pointerEvents(events, false) 載入筆跡座標

    convert(null, DIGITAL_EDIT) ← 非常關鍵的一步！指示引擎將手寫筆跡辨識並轉換為標準印刷體字形 (Typeset Glyphs)。文字辨識不做此步驟。

  5. 準備字型與畫布

    • 從 assets/fonts/ 載入 stix_regular.otf
    • 建立 TypefaceMap，將字型註冊為 "STIX" key（對應上面的 CSS）
    • 傳給自定義的 ImagePainter

  6. 雙重 Export

    ① editor.export_(null, filePath, MimeType.PNG, imagePainter) → 引擎接管 Android Canvas，使用 STIX 字型渲染成透明背景 PNG（存在快取目錄，如 math_render_1234.png）

    ② editor.export_(null, MimeType.LATEX) → 匯出 LaTeX 字串（備用，例如 x=\frac{-b\pm\sqrt{b^2-4ac}}{2a}）

  7. 回傳串接格式

    " /data/.../cache/math_render_1234.png|||x=\frac{-b\pm\sqrt{b^2-4ac}}{2a} "

    大結果（>5000 字元）則寫入暫存 JSON 檔案，回傳檔案路徑。

階段三：使用者確認介面 (nNoteAS Dialog)

  1. 收到字串後，用 ||| 切割出 pngPath 和 latexString
  2. 彈出 OcrMathBoxEditorDialogFragment 預覽視窗：
    • 用 BitmapFactory.decodeFile(pngPath) 將 PNG 載入 ImageView
    • 隱藏原本的 KaTeX/LaTeX 輸入框和鍵盤，只顯示精美數學圖片
    • 提供 SwitchCompat 控制 Math Solver 開/關（切換後重新辨識）
  3. 使用者選擇：
    • 取代筆跡（Replace）：傳 PNG 路徑，移除原始手寫筆跡
    • 保留筆跡（Keep）：傳 PNG 路徑，保留原始手寫筆跡

階段四：寫入畫布與儲存 (nNoteAS Canvas)

Dialog 關閉後，進入 createMathGraphicsImage()：

  1. 讀取與計算尺寸 (Aspect Ratio)

    用 BitmapFactory.Options.inJustDecodeBounds 讀取 PNG 真實長寬。寬度設定為原筆跡的 1.5 倍（更好的閱讀體驗），依長寬比計算高度。

  2. 位置偏移 (Offset)

    X 座標對齊原筆跡，Y 座標設在原筆跡下方 20px，避免直接重疊。

  3. 實體檔案轉移 ⚠️ 非常重要

    原 PNG 在 nOCR 快取目錄，可能被系統刪除。透過 UUID 命名，將圖片複製到筆記本專屬的 BookDirectory，確保匯出筆記時圖片不丟失。

  4. 嵌入原始筆跡（為雙擊還原準備）

    + mathImage.isMathImage = true;
+ mathImage.setOriginalHandwriting(cloneStrokes);

    Clone 原始 selectedStrokes 的副本存入 GraphicsImage，序列化時一併寫入儲存。

  5. Undo/Redo 整合

    建立 new GraphicsImage(transform, x, y)，呼叫 setFile() 綁定檔案，送入 Undo/Redo 歷史紀錄。Canvas 透過 drawBitmap 顯示。

新舊作法對比

假設使用者在畫布上手寫了 ½（二分之一）：

❌ 舊作法 KaTeX 繞遠路

  1. 引擎把手寫變成 LaTeX：\frac{1}{2}
  2. nNoteAS 辛苦地開一個 WebView
  3. 載入 KaTeX 的 JS/CSS 函式庫
  4. 拜託 KaTeX 翻譯機把字串畫出來
  5. 截圖貼到畫面上，字體普通

問題：慢、耗效能、WebView 肥大、排版有時出錯

✅ 新作法 Native PNG 直通車

  1. 引擎直接辨識手寫
  2. convert(DIGITAL_EDIT) 轉為印刷體
  3. 拿出 STIX 字型，內部渲染
  4. 直接存成一張教科書級 PNG
  5. nNoteAS 把它當圖片貼上畫布

好處：完全解耦、不需要 WebView、速度快、品質高

💡 核心哲學：nNoteAS 不需要知道怎麼畫數學符號，它就只是「顯示一張圖」。所有複雜的排版與字型處理，都交給本來就很擅長這件事的 MyScript 原生引擎（在 nOCR 裡）。

雙擊還原功能

問題背景

數學辨識將手寫筆跡轉為 PNG 圖片後，使用者無法編輯或擦除公式中的部分內容。需要一個「後悔藥」機制：雙擊數學圖片時，圖片消失，恢復原始手寫筆跡。

實作步驟

步驟修改檔案做什麼
1 GraphicsImage.java 新增兩個欄位：
boolean isMathImage — 標記是否為數學辨識產生的圖片
List<Graphics> originalHandwriting — 存放原始筆跡副本
Protocol v2 序列化支援新欄位
2 HandwriterView.java
createMathGraphicsImage()		 建立 Math Image 時先 clone 筆跡再存入： 
ArrayList<Graphics> cloneStrokes = new ArrayList<>();
for (Graphics g : selectedStrokes) {
    if (g instanceof Stroke)
        cloneStrokes.add(((Stroke) g).getCloneGraphics());
}
mathImage.isMathImage = true;
mathImage.setOriginalHandwriting(cloneStrokes);
3 HandwriterView.java
GestureListener		 手指雙擊還原（唯一入口）： 
@Override
public boolean onDoubleTap(MotionEvent e) {
    for (GraphicsImage img : pageLayer.images) {
        if (img.isMathImage && img.getOriginalHandwriting() != null) {
            if (img.getBoundingBox().contains(tapX, tapY)) {
                page.removeImage(img);          // 1. 移除圖片
                for (Graphics s : img.getOriginalHandwriting()) {
                    page.addStroke((Stroke) s);  // 2. 恢復筆跡
                }
                nDrawHelper.aDrawClearCanvas();  // 3. 重繪
                return true;
            }
        }
    }
    return false;
}
4 HandwriterView.java
strokesToGraphics()		 修復重繪問題：確保還原的筆跡能正確渲染在 Canvas 上
5 HandwriterView.java
remove_for_erase()		 修復橡皮擦行為：確保數學圖片被擦除時正確處理
6 OcrMathBoxEditor
DialogFragment		 移除不需要的 Math Dialog 插入按鈕（簡化 UI）
觸控筆雙擊和橡皮擦不會觸發還原，維持原本的 zoomOut / 擦除行為。只有手指雙擊觸發。
E-ink 裝置上，手指事件走 GestureDetector，不走 TouchHandler。

序列化格式（Protocol v2）

Protocol v1: uuid, 座標, constrainAspect
Protocol v2: v1 的所有欄位 + isMathImage + originalHandwriting 筆跡數量 + 每條筆跡的完整資料

writeToStream() 和 DataInputStream 建構子支援讀寫這兩個新欄位，向後相容：v1 格式的檔案讀入時 isMathImage 預設為 false。
