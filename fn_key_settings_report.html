3. 為什麼改 PhoneWindowManager

PhoneWindowManager 的角色

PhoneWindowManager 是 WindowManagerPolicy 介面的實作，是 Android 系統中所有實體按鍵的策略決策中心。它決定了每一個硬體按鍵按下後該執行什麼動作。

✅ 選擇這裡修改的原因

  • 系統級攔截：在 Policy 層攔截意味著 FN 鍵無論在任何 App、任何畫面（包括 Launcher、Keyguard）都會生效
  • 已有成熟模式：KEYCODE_SETTINGS、KEYCODE_HOME 等按鍵都在此處理，FN 鍵理所當然也在這裡
  • 已有 showSystemSettings()：此方法已存在（line 2065），呼叫 Settings.ACTION_SETTINGS，無需額外新增程式碼
  • Event 消費：return true 會消費掉事件，App 不會再收到此 keycode，避免衝突

❌ 為什麼不在其他地方改

方案缺點
改 App 層（在某個 Activity 裡監聽）只有該 App 在前台時才有效；其他 App 或 Launcher 上無效
改 Key Layout (.kl) 映射.kl 只能映射到其他 keycode，無法直接觸發啟動 Activity
改 InputManagerService太底層，不適合放業務邏輯；PhoneWindowManager 才是設計好的策略層
用 AccessibilityService需要額外安裝 App、使用者要手動開啟無障礙權限，不可靠

5. 修改細節與差異比較

PhoneWindowManager 有兩條按鍵攔截路徑：Old（傳統）和 New（KeyGestureEvent 新架構），我們在兩條路徑都加了 KEYCODE_FUNCTION 處理。

兩條路徑的差異

項目Old PathNew Path
判斷方式firstDown（方法開頭已計算好的 boolean）手動檢查 ACTION_DOWN && repeatCount==0
原因Old 方法有 firstDown 區域變數New 方法沒有此變數，需自行判斷
效果完全相同 — 僅在首次按下時觸發一次 Settings

已存在的 showSystemSettings() 方法

📄 PhoneWindowManager.java — line 2065
private void showSystemSettings() {
    startActivityAsUser(new Intent(android.provider.Settings.ACTION_SETTINGS),
            UserHandle.CURRENT_OR_SELF);
}

呼叫 Settings.ACTION_SETTINGS 等效於你提供的 onSettingsClick() 中的 ComponentName("com.android.settings", "com.android.settings.Settings")，但更通用（會走系統預設的 Settings App resolve）。

6. 驗證方式

步驟

  1. 重新編譯 framework（make services 或整包 make systemimage）
  2. 將產出的 services.jar 推送至裝置 /system/framework/，或刷入新的 system image
  3. 重啟裝置
  4. 測試：
    • 按下 FN → 應開啟 Settings App
    • 按下 HOME → 應回到桌面（不受影響）
    • 按下 POWER → 應正常關螢幕/喚醒（不受影響）
  5. 觀察 Logcat：adb logcat | grep -i "PhoneWindowManager\|showSystemSettings\|KEYCODE_FUNCTION"
