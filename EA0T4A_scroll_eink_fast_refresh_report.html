捲動 E-Ink 快速刷新整合報告

讓全系統「上下捲動」時自動切換 E-Ink 快速波形，手指放開即復原
範圍：framework.jar
異動檔案：2 files
涵蓋：ScrollView / ListView / RecyclerView / WebView
日期：2026-06-12

▎一句話總結

延續 SeekBar 那套「動作時進快刷、停止時復原」的概念，這次把它擴大到整個系統的捲動：使用者上下捲動任何清單／頁面時切到 E-Ink 快速波形讓捲動順暢，手指一放開就切回正常高畫質波形。只動 2 個 Framework 檔案，全系統捲動（含 Settings All apps）自動受惠。

※ E-Ink 波形原理、廣播參數意義已於《SeekBar 拖動 E-Ink 快速刷新報告》說明，本報告不重複，聚焦在「捲動」的特殊處理。

1 先驗知識：捲動為什麼比 SeekBar 棘手

▎難點一：「可捲動」其實是好幾種不同類別

Android 沒有單一的「捲動元件」，而是分散在多個類別，各自用不同方式捲動：

類別用在哪怎麼捲
ScrollView / NestedScrollView一般可捲頁面改 scrollY
ListView / GridView舊式清單（聯絡人、檔案）位移子 view
RecyclerView現代清單，Settings All apps 就是這個位移子 view
WebView / TextView網頁、長文字改 scrollY
⚠️ RecyclerView 的原始碼不在這棵 tree 裡（只有預編譯 AAR）。所以沒辦法像 SeekBar 那樣直接改它的原始碼——這是本次最大的技術障礙。

▎難點二：捲動沒有明確的「結束」事件

SeekBar 有乾淨的「放開（ACTION_UP）」當結束點；但捲動是連續事件流，系統不會通知你「捲完了」，尤其慣性滑動（fling）放手後還會自己滑一段。所以「何時退出快刷」需要另外設計判定方式。

2 解法：在 View 共同基底層統一攔截

關鍵發現：上面所有捲動元件，捲動時都會呼叫繼承自 View 的同一個方法。所以不必逐一改每個類別，只要攔 View 這一層，就能涵蓋全部——連原始碼不在 tree 裡的 RecyclerView 也一起涵蓋（因為它呼叫的是繼承來的 View 方法）。

攔截點涵蓋元件作用
View.awakenScrollBars()ListView / GridView / RecyclerView偵測捲動 → 進快刷
View.onScrollChanged()ScrollView / WebView / TextView偵測捲動 → 進快刷
ViewRootImpl.processPointerEvent()全部（視窗唯一輸入入口）抓 ACTION_DOWN/UP → 手指放開即復原
為什麼「手指放開」要放在 ViewRootImpl？因為 RecyclerView 捲動時，子 view 會先收到 CANCEL、真正的 UP 是送到容器；只有在視窗層唯一的輸入入口 ViewRootImpl 才能可靠且只抓一次真正的手指放開。

▎核心邏輯（簡化）

// View.java — 偵測捲動時（手指按著才進、每手勢一次） ntxEinkReportScroll() { if (!手指按著 || 已在快刷) return; 已在快刷 = true; sendBroadcast(GLOBAL_REFRESH, waveformmode=1, flags=0x2000, dithermode=0x301); } // ViewRootImpl.java — 手指放開時復原 DOWN → 手指按著 = true UP/CANCEL → if (已在快刷) { sendBroadcast(GLOBAL_REFRESH, waveformmode=0); 清狀態; }
5 設計考量
  • 手指按著才進快刷：避免「程式自動捲動 / 動畫」誤觸發，也防止沒有對應放開事件而卡死。
  • 每手勢只發一次 enter：捲動事件極頻繁，用狀態旗標確保不會狂發廣播。
  • 放手即復原（含 CANCEL）：來電、鎖屏等中斷會收到 CANCEL，一樣會復原，不會卡在快刷。
  • 不改 RecyclerView 原始碼：靠攔截它繼承的 View.awakenScrollBars()，所以原始碼不在 tree 裡也能涵蓋。
  • 慣性 fling 不快刷：依拍板選擇「手指放開即復原」，放手後的滑動段維持正常波形。
6 如何驗證廣播有發

接收端 PowerManagerService 收到廣播時會印 log，最快的驗證：

adb logcat | grep "SMode"
動作預期 log
進快刷(SMode) ... Globle Mode - updatemode(0) wait(0) waveform(1) flags(2000) dithermode(301)
復原(SMode) ... Globle Mode - ... waveform(0)

8 重點觀察
  • 不該卡在快刷：任何操作後，靜止畫面應回到正常高畫質波形（不是一直黑白快刷）。用 grep "SMode" 確認每個 enter 後最終都有對應的 waveform(0)。
  • RecyclerView 是關鍵驗證點：它原始碼不在 tree 裡，靠 View.awakenScrollBars() 這層攔截——所以「Settings All apps 捲動能進快刷」是這次改動成不成立的指標。
若有實機，建議優先測兩項：① Settings All apps 捲動是否進快刷；② 靜止後是否回到正常波形。
