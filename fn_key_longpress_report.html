FN Key 長按開啟 Settings
實作技術報告

📅 2026-04-07 📄 PhoneWindowManager.java 🔖 de5eb649791e [EMI Test]

🎯 總覽

將 FN 實體按鍵的 開啟 Settings 功能 從「短按」改為「長按」觸發。並且修正了原本的長按偵測機制 ── 從「放開後才判斷按住時長」改為「按住滿 1 秒立刻觸發」。

操作舊行為新行為
短按 < 1s 開啟 Settings ✨ + 前光 Toggle 前光 Toggle（開/關）💡
長按 ≥ 1s （僅 Log，無動作） 開啟 Settings ⚙️（按住即觸發）

🔧 改動說明

① 移除短按開啟 Settings（2 處）

原本在 interceptSystemKeysAndShortcutsOld 和 interceptSystemKeysAndShortcutsNew 中都有 KEYCODE_FUNCTION 的 case，短按便會呼叫 showSystemSettings()。這兩處都已移除：

interceptSystemKeysAndShortcutsOld / New — 移除的程式碼
  //20260401 Anson For Emi Test
  case KeyEvent.KEYCODE_FUNCTION:
      if (firstDown) {
          showSystemSettings();
      }
      return true;

② 長按觸發 Settings

在 onFunctionKeyLongPress() 中加入 showSystemSettings()：

onFunctionKeyLongPress()
  private void onFunctionKeyLongPress() {
      Log.d(TAG, "FUNCTION key LongPressed");
      showSystemSettings();  // ← 新增：長按開啟 Settings
  }

③ 核心改動：Handler.postDelayed 長按機制

這是最重要的改動 — 將長按偵測從「放開後判斷時長」改為「按住滿時間立刻觸發」。

新增的成員變數
  private boolean mFunctionKeyLongPressTriggered = false;
  private final Runnable mFunctionKeyLongPressRunnable = () -> {
      mFunctionKeyLongPressTriggered = true;
      onFunctionKeyLongPress();
  };
新的 KEYCODE_FUNCTION 處理邏輯（interceptKeyBeforeDispatching 內）
if (keyCode == KeyEvent.KEYCODE_FUNCTION) {
    final long now = SystemClock.uptimeMillis();

    if (action == KeyEvent.ACTION_DOWN && event.getRepeatCount() == 0) {
        // debounce on DOWN
        if (now - mLastFunctionKeyTriggerTime < DEBOUNCE_DELAY_MS) {
            return keyConsumed;
        }
        mFunctionKeyPressStartTime = now;
        mFunctionKeyLongPressTriggered = false;
        mHandler.removeCallbacks(mFunctionKeyLongPressRunnable);
        mHandler.postDelayed(mFunctionKeyLongPressRunnable, LONG_PRESS_TIMEOUT_MS);
        return keyConsumed;

    } else if (action == KeyEvent.ACTION_UP) {
        mHandler.removeCallbacks(mFunctionKeyLongPressRunnable); // 取消計時

        if (mFunctionKeyPressStartTime == 0) {
            return keyConsumed;
        }
        mLastFunctionKeyTriggerTime = now;

        // 長按已觸發? 跳過短按
        if (!mFunctionKeyLongPressTriggered && !event.isCanceled()) {
            onFunctionKeyShortPress();
        }

        mFunctionKeyPressStartTime = 0;
        return keyConsumed;
    }
    return keyConsumed; // 吃掉 repeat > 0 的 DOWN
}

💡 核心問題：為何長按不再需要等 UP？

🔑 關鍵答案

長按仍然有 DOWN 和 UP 事件 — 硬體一定會產生 DOWN/UP。差別在於：舊邏輯在 UP 才判斷，新邏輯在 DOWN 就啟動計時器，計時器到期自動觸發，不需要等 UP。

⚙️ Handler.postDelayed 機制詳解

Android 的 Handler 是一個訊息佇列機制，可以將任務排程到主執行緒上延遲執行。這個做法在 AOSP 中被廣泛使用（例如 Power 鍵長按、Home 鍵長按都是同樣的模式）。

💡 為什麼不需要等 UP？

因為計時器是獨立運行的。 按下 DOWN 的那一刻，一個延遲 1 秒的 Runnable 就被丟進了 Handler 的 MessageQueue。這個 Runnable 會自己在 1 秒後被 Looper 取出並執行 — 完全不需要任何其他按鍵事件來觸發它。

如果使用者在 1 秒內放開（產生 UP），我們就呼叫 removeCallbacks 把那個 Runnable 從佇列中刪掉，然後走短按邏輯。

如果使用者持續按住超過 1 秒，Runnable 就會自動執行 — UP 來了也只是「清理」，不會重複觸發。

三個關鍵 API

API時機作用
postDelayed(runnable, 1000) ACTION_DOWN 排程：1 秒後自動執行 onFunctionKeyLongPress()
removeCallbacks(runnable) ACTION_UP 取消：如果提前放開，把排程中的 Runnable 移除
mFunctionKeyLongPressTriggered FLAG 互斥：確保長按觸發後，UP 不會再觸發短按

💡 短按流程

1

使用者按下 FN → 系統發送 ACTION_DOWN, repeat=0

2

記錄 mFunctionKeyPressStartTime = now，設定 mFunctionKeyLongPressTriggered = false

3

呼叫 mHandler.postDelayed(runnable, 1000) → 排程長按

4

使用者在 1 秒內放開 → ACTION_UP

5

removeCallbacks 取消排程中的長按 Runnable

6

mFunctionKeyLongPressTriggered == false → 呼叫 onFunctionKeyShortPress() → 前光 Toggle 💡

⚙️ 長按流程

1

使用者按下 FN → ACTION_DOWN, repeat=0

2

記錄時間 + postDelayed(runnable, 1000)

3

使用者持續按住... ⏱️ 1 秒到達！

4

Handler 自動執行 Runnable → mFunctionKeyLongPressTriggered = true → onFunctionKeyLongPress() → 開啟 Settings ⚙️

5

使用者放開 FN → ACTION_UP

6

removeCallbacks（Runnable 已執行，無影響）+ mFunctionKeyLongPressTriggered == true → 跳過短按 ✓
