一、執行摘要

本報告評估在現有 批次辨識 (Batch) 架構基礎上，新增增量辨識 (Incremental Recognition) 功能的可行性與開發時程。

結論：技術上完全可行，現有架構（AIDL、MyScript iink SDK）已具備支撐增量辨識的基礎能力。預估需 5–7 個工作天，由 1 名開發人員完成，包含設計、實作、測試與整合。

二、現況分析

現有系統架構

Client 端LimeIME（輸入法 App）— 提供手寫畫布，收集筆跡
Server 端nOCR（辨識引擎 Service）— 內嵌 MyScript iink SDK
通訊方式Android AIDL (Binder IPC) 跨程序通訊
辨識模式Batch 僅支援批次辨識

現有 Batch 流程

使用者寫完所有筆跡 → 等待 idle timeout → 將全部筆跡序列化為 JSON → 同步 AIDL 送到 nOCR → 建立 Editor → 辨識 → 銷毀 Editor → 回傳結果

現有方案的痛點

  • ⏳ 延遲高：需等使用者停止書寫 + 完整辨識時間，體感等待 1-2 秒
  • 🔒 UI 阻塞：辨識期間畫布和候選欄鎖定，無法繼續操作
  • 🔁 資源浪費：每次辨識都重建/銷毀 Editor，產生不必要的初始化開銷
  • 📉 互動性差：使用者無法在書寫過程中即時看到辨識結果

三、目標需求

需求描述

在保留現有 Batch 功能的前提下，新增 Incremental 增量辨識模式，讓使用者邊寫邊看到辨識結果，提升即時互動體驗。

預期效果

項目現況 (Batch)目標 (Incremental)
辨識觸發寫完後 idle timeout每筆離開 (ACTION_UP) 即觸發
結果延遲1–2 秒< 300ms

模式切換—提供 Toggle 按鈕，可隨時切換

四、可行性分析

MyScript iink SDK 支援度

✅ SDK 原生支援增量模式

MyScript iink 的 Editor.pointerEvents() 方法提供第二參數：

  • pointerEvents(events, false) — 取代模式（現有 Batch 使用）
  • pointerEvents(events, true) — 累加模式（Incremental 將使用）

不需要額外購買授權或升級 SDK 版本，現有 SDK 已完整支援。

AIDL 通訊可行性

✅ AIDL 支援 oneway 非阻塞呼叫

Android AIDL 原生提供 oneway 修飾詞，可實現非阻塞的跨程序事件傳輸，完美適合高頻觸控事件的即時傳遞。

需要新增的技術元件

元件說明複雜度
InkPointerEvent跨程序傳輸觸控事件的 Parcelable 類別低
IOcrCallback.aidlServer → Client 非同步推送結果的回呼介面低
OcrAidl 擴充新增 beginSession、feedPointerEvents、endSession 方法中
IncrementalSession維持 MyScript Editor 持久化存活的 Session 封裝中
SessionManagerSession 池管理：並行數限制、idle 逾時回收中
LimeIME 整合觸控事件轉換、callback 處理、UI 候選欄更新高

五、架構設計方案

核心設計原則

  • 向後相容：Batch 模式完全保留，Incremental 為新增功能
  • 非阻塞通訊：使用 oneway AIDL 發送觸控事件，UI 不卡頓
  • 非同步結果推送：透過 IOcrCallback 從 nOCR 推回辨識結果
  • Session 持久化：Editor 在 Session 存活期間不銷毀，支援筆跡累加
  • 自動資源回收：idle Session 超時自動清理，避免記憶體洩漏

Batch 現有架構

LimeIME → 同步 AIDL → nOCR
建立 Editor → 辨識 → 銷毀 Editor
結果直接 return

Incremental 新增架構

LimeIME → oneway AIDL → nOCR
持久化 Editor（Session 管理）
結果透過 IOcrCallback 推回

六、開發時程（5–7 天）

Session 管理 + MyScript Editor 持久化

Day 3 — LimeIME Client 端整合（上）

七、風險評估

風險項目等級說明應對策略
MyScript Editor 記憶體洩漏 中 持久化 Editor 若未正確關閉，可能導致記憶體持續增長 SessionManager 強制 idle 逾時回收 + DeathRecipient 重建機制
高頻 AIDL 呼叫效能 中 每個觸控 MOVE 事件都跨程序傳輸，可能有 Binder 瓶頸 使用 oneway 非阻塞 + InkPointerEvent[] 批量傳輸（含歷史事件）
nOCR 程序崩潰 低 辨識引擎異常退出，Session 丟失 IBinder.DeathRecipient 自動重新綁定，下次書寫時重建 Session
Batch 功能回歸 低 新增 Incremental 可能影響原有 Batch 流程 兩模式完全獨立路徑，不修改 Batch 相關程式碼
E-ink 顯示延遲 中 E-ink 螢幕刷新率低，即時候選字更新可能不明顯 測試階段評估最佳更新頻率，必要時調整辨識觸發時機
