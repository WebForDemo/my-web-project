02 / 09 根本原因

vendor 的 build 時間被凍結在舊日期

ro.vendor.build.date.utc 不是來自 source 日期,而是 build 當下從 out/build_date.txt 抓的時間,寫進 build.prop。

① 時間來源

out/build_date.txt 第一次 build 抓一次,之後 incremental 一直沿用——這是刻意設計,讓增量編譯時間穩定。

② build.prop

分區的 build.prop 只在「不存在時」才產生一次。已存在就被當 up-to-date,不重產。

③ 結果

兩者都沿用舊的 → vendor 時間被釘死在「上次 clean build 的日期」,無論怎麼增量重編都不變。

一句話:incremental build 不會更新 vendor 的 build 時間,它停在最後一次乾淨編譯那天。

04 / 09 試過但無效

為什麼那些手段都沒用

A 手動改 build_date.txtbuild.prop 早已存在、被當 up-to-date,新時間從沒被讀到。 ✕ 無效
B repo sync 最新 sourcesync 只動 source,完全不碰 out/ 裡的舊產物。 ✕ 無效
C build.sh 自動刪 build.prop改了沒重新 source,舊函式照跑;且 split-build 不會重產 build.prop。 ✕ 無效
D 打包加 --override_timestamp此參數只走 incremental OTA 路徑;你的是 full payload,對 per-partition 時間不生效。 ✕ 無效
05 / 09 真正有效

砍掉整個 out 重 build ✓

關鍵在於它同時強制了「時間重抓」與「build.prop 重產」——前面每個單點手段各缺一半,這個一次補齊。

out 清空後發生什麼

build_date.txt 不存在 → 重抓當下時間。三個 build.prop 也不存在 → 重新產生並讀到新時間。

結果

→ 1782213174

三個分區全變 2026-06-23 19:12,新於裝置的 6/12 → 防降級檢查全過。

代價:等於全 clean,重編較久。輕量等效 = 出 OTA 前 rm out/build_date.txt + 那三個 build.prop(尚未驗證,需測)。
06 / 09 另一條路 · 雞生蛋

改 update_engine 繞過檢查

這兩筆 commit 直接關掉降級檢查,改法正確——但它必須已經跑在裝置上才生效。

b12442c0  delta_performer.cc  CheckTimestampError()  →  return kSuccess
2abad906  hardware_android.cc  AllowDowngrade()  →  return true
雞生蛋　驗證 OTA 的是裝置「當前 slot 上正在跑」的 update_engine。失敗 log 裡沒有出現新加的 Skip partition version downgrade check. → 代表裝置跑的還是舊的、沒改過的版本。你沒辦法靠 OTA 把「能放行 OTA 的 update_engine」送進去。

解:先用 fastboot 刷一次含這兩筆 commit 的版本進去,之後 OTA 才會 bypass。

08 / 09 結論與後續

兩條獨立的「能更新」理由

  • 理由 A · timestamp 夠新 — build 時間 > 裝置現有版本就會過,跟 update_engine 有沒有改無關。nuke-out 那次成功,靠的是這個。
  • 理由 B · 裝置已跑 bypass 版 — 與 timestamp 無關,但前提是含 commit 的版本已在裝置上。成功更新並重開後,這台已「轉正」。

日後出 OTA · 防呆

要出能蓋過舊機的 OTA → 確保 vendor 時間新於目標機(最穩:出包前清 out)。
