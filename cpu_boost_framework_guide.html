CPU Boost Broadcast 架構講解

App / ADB 廣播觸發 Framework，再由 init.rc 切換 CPU governor
frameworks/base/services/core/java/com/android/server/power/PowerManagerService.java
1 目標與定位

這次改法的核心目標，是把原本只能靠 adb shell 手動操作的 CPU governor 切換，做成一條可由 任意 app 或 adb am broadcast 直接觸發的系統流程。

  • 輸入端：App 或 ADB 發送 broadcast
  • Framework 入口：PowerManagerService 內新增 CpuBoostReceiver
  • 中介狀態：寫入 sys.ntx.book.cpu_boost
  • 執行端：vendor init.mt8171.rc 監聽 property 並寫入 governor

3 為什麼走 Property + init.rc

一開始如果直接讓 framework 去寫 /sys/devices/system/cpu/.../scaling_governor，會碰到 SELinux 與分層問題。這次最後採用的是：

Framework 只負責決策
→
System Property 傳遞狀態
→
init 以 root 寫 sysfs

這樣做的好處是：

  • framework 不直接碰 sysfs
  • 真正寫 governor 的仍然是 init
  • vendor 端可以自由調整 policy0 / policy6 或之後改成其他策略

補充：什麼是 sysfs？什麼是 SELinux？

sysfs 可以把它理解成 Linux kernel 對外暴露的一組虛擬檔案介面。像這次用到的 /sys/devices/system/cpu/cpufreq/policy0/scaling_governor，本質上不是一般設定檔，而是 kernel 提供的控制入口；對它做 write，就等於在改 CPU governor 行為。

SELinux 則是 Android 上的強制存取控制機制。就算某個 process 有能力呼叫 Java API 或 shell 指令，也不代表它一定能寫 /sys 或某個 property。這次之所以不讓 framework 直接寫 sysfs，就是因為 system_server 直接碰 vendor / kernel 控制點，很容易被 SELinux 的 neverallow 擋住。改成 Framework → sys property → init.rc → sysfs，就是把權限邊界分清楚。

7 為什麼從 vendor.* 改成 sys.*

中途有一個重要修正：原本一度想用 vendor.ntx.book.cpu_boost，但 build 時被 sepolicy 的 neverallow 擋下來，原因是 system_server 不能直接 set vendor property。

vendor.ntx.book.cpu_boost
✕
sys.ntx.book.cpu_boost

所以最終方案改為讓 framework 寫 sys.*，再由 vendor init.rc 去監聽這個 system property。這是這次能通過 build 的關鍵點。

8 這次卡住的來龍去脈

這次中間之所以會失敗，不是因為 Java 寫法錯，而是因為 property 的分層碰到 SELinux 規則。簡單講：

想讓 system_server 寫
vendor.ntx.book.cpu_boost
→
被 neverallow 規則判定不合法
→
改成寫
sys.ntx.book.cpu_boost

neverallow 的意思不是「現在沒開權限」，而是「政策上禁止你這樣設計」。所以就算另外補 allow，編譯 policy 時還是會被打掉。

檔案 它在管什麼 這次跟它的關係
/mnt/aosp2/android16_mtk_system_b0mp5_aiv/system/sepolicy/private/property.te 定義 property 類型與 property 相關的 neverallow 規則 這次真正卡住 build 的核心規則就在這裡，限制 coredomain 亂設 vendor property。
/mnt/aosp2/android16_mtk_system_b0mp5_aiv/system/sepolicy/private/property_contexts 把 property 名稱前綴對應到 SELinux type vendor.* 預設會被歸到 vendor property 類型，所以一開始才會踩線。
/mnt/aosp2/android16_mtk_system_b0mp5_aiv/system/sepolicy/private/system_server.te 定義 system_server 可以做哪些事 如果真的有某個合法 property type 要給 framework 寫，通常會在這裡補 set_prop(system_server, ...)。
/mnt/aosp2/android16_mtk_system_b0mp5_aiv/system/sepolicy/private/shell.te 定義 adb shell 可以做哪些事 如果想靠 adb shell setprop 直接測 property，要看這裡有沒有放行。

9 優缺點與後續建議
優點 ─ 對接快速、ADB 好測、邏輯集中
目前狀態 ─ 偏驗證版，不是最終量產安全策略
風險 ─ 任意 app 都可送 broadcast
  • 如果只是 bring-up 或內部驗證，這版已經足夠直接。
  • 如果要量產，建議把 receiver 改回受保護的 permission，避免第三方 app 任意切 governor。
  • 未來若要更 Android 正規，可再收斂成受保護 system API 或 vendor service。
