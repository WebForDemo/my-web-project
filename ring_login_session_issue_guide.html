Acer Ring 登入 / Session 問題講解

用白話說明 Cognito、Backend Token、Session、HTTPS 憑證，並針對這次問題畫出實際架構圖
app/src/main/java/com/hukuibio/afsring/data/repository/AuthRepository.java / app/src/main/java/com/hukuibio/afsring/api/RetrofitClient.java
1 先講結論

這次登入原本會失敗，不是 API Key 錯，也不是 route 錯，也不是 body 格式錯。真正的主問題是：

  • App 先向 Cognito 取 Backend Token，這一段本來就能通。
  • 接著 App 再向 Ring Backend 建立 Session，這一段原本因為 HTTPS 憑證不被 Android 信任，所以失敗。
  • 你用 curl -k 可以成功，是因為 -k 代表「忽略憑證驗證」。
  • 我後來把流程拆成兩條 client：Cognito 繼續走原本安全方式，只有 Ring Backend 的 Session 先走暫時測試方式，所以最後才通。

一句話版本

Cognito 沒問題，Ring Backend 的 HTTPS 才是問題點。 App 之所以後來成功，是因為只有去拿 Session 的那條連線被改成暫時放寬憑證檢查。

2 名詞先翻白話
名詞 白話解釋
Acer SSO Acer 的登入入口。使用者在瀏覽器輸入帳密後，最後會把結果導回 App。
Cognito Amazon 提供的身分驗證服務。這個專案用它來拿某些登入相關 Token。
Backend Token 後端系統自己要用的 Token。不是給使用者看的，而是 App 先去幫後端拿的一把鑰匙。
User Access Token 代表使用者身分的 Token。現在這個專案暫時是把 deep link 回來的 code 直接當成它來用。
Session 後端核對完身份之後，發一個短期可用的工作證給 App。之後很多 Ring API 都靠這個 sessionId 存取。
HTTPS 憑證 可以理解成網站的「身分證」。App 連線時，會先檢查對方是不是它聲稱的那台伺服器。
Trust Anchor Android 信任的憑證根。當 log 出現 Trust anchor for certification path not found，意思是 Android 不相信這張憑證鏈。
TLS Handshake HTTPS 真正開始傳資料前，雙方先做的安全握手。握手沒過，HTTP request 就算印在 log，也不代表 server 真正處理到了。
curl -k -k 代表忽略憑證檢查，所以常用來測「API 本身是否活著」，但不代表 App 的正式 HTTPS 一定也能過。

4 原本為什麼失敗

失敗點其實在第 4 步

Acer SSO
→
Cognito 拿 Backend Token
→
Ring Backend 建 Session
→
後續 API

原本 log 裡一直出現 Trust anchor for certification path not found，這句話的白話是：

  • Android 嘗試連到 https://123.51.250.96:1500
  • Server 回給 Android 一張 HTTPS 憑證
  • Android 檢查後發現「我不信這張憑證鏈」
  • 所以直接中止連線，根本還沒進到真正的 API 邏輯

為什麼 curl 看起來能過

你用的指令是 curl -k ...。這個 -k 的意思是：不要驗證對方憑證。

所以它只能證明：

  • 這個 IP 上的 API server 有活著
  • Route / Header / Body 沒有寫錯

它不能證明 Android 用標準 HTTPS 驗證也一定能過。

6 這次程式怎麼改才通

這次不是把所有登入流程都改成不安全，而是只拆出 Ring Backend 這條：

連線用途 現在走哪條 client 目的
Cognito Backend Token 標準 client 保持原本安全驗證，不碰 Acer SSO / Cognito 這段
Ring Session 暫時測試 client 讓目前這個 https://123.51.250.96:1500 能先測通
Ring Data API 暫時測試 client 之後後續 Ring API 也能延續同樣的測試方式

實際程式對應

  • AuthRepository.login() 內的 authService.getBackendToken(...)：還是原本方式
  • AuthRepository.createSession() 內的 ringAuthService.createSession(...)：改成新的 Ring backend client
  • RetrofitClient.getAuthService()：標準 client
  • RetrofitClient.getRingAuthService()：暫時測試 client

  • 11:34:25 Cognito 回 200，代表這段成功
  • 11:34:25 App 接著向 Ring Backend 建 Session

  • 11:34:29 App log 顯示 Session Created. ID: ...

這代表目前調整後的登入流程，至少在你現在這個測試環境下，已經完整走通。

8 現在的風險與之後正規作法

現在的風險

現在能通，是因為 Ring Backend 這條先放寬了憑證檢查。這適合測試，但不適合正式版。

  • 正式版不該長期忽略憑證驗證
  • 現在打的是 IP，不是正式 HTTPS domain
  • 之後後端給正式 domain 和完整憑證後，要把這段測試寫法撤掉

正規作法

  • 後端提供正式的 HTTPS domain
  • 憑證鏈完整，Android 可以正常信任
  • App 改回統一標準 HTTPS 驗證
  • 不再需要暫時測試 client
9 你可以怎麼用這份說明

這份說明適合拿來做三件事：

  • 自己回頭理解這次到底卡在哪一層
  • 跟後端說明「現在不是 API 邏輯錯，而是 HTTPS 憑證 / IP 入口問題」
  • 之後正式上線時，提醒團隊把暫時測試 client 撤掉
這份頁面描述的是「目前測試環境如何先打通」的實際狀態，不代表最終正式版安全方案。
