FrontLight 底層串接分析

Settings key → system_server → lights / HAL 的完整追蹤
frameworks/base + vendor/mediatek/proprietary/packages/apps/MtkSettings
1 結論先看
  • screen_brightness 被拿來當 front light 白光值，但它同時也是 Android 標準亮度 key，所以和一般 display brightness 流程有耦合。
  • screen_brightness_color、screen_front_light_mode 是後續 FrontLight 客製新增到 Settings.System 的 key。
  • 整體有兩條路徑：一條寫進 Settings.System 供 observer 消費，一條由 UI 直接 call PowerManager 立即生效。
  • screen_front_light_mode 是跨層 gating 開關，UI、PowerManagerService、LightsService 都會看它。
  • 色溫通道目前借用 LIGHT_ID_ATTENTION，不是獨立的 light type。
2 三個 Key 與角色
Key 定義 主要寫入者 主要讀取者 影響
screen_brightness Settings.System settings put、FrontLightPreference PowerManagerService、標準 display brightness 流程 front light 白光亮度，且與系統亮度共用 key
screen_brightness_color vendor patch settings put、FrontLightPreference PowerManagerService、FrontLightPreference front light 色溫通道亮度
screen_front_light_mode vendor patch settings put、FrontLightPreference PowerManagerService、LightsService、FrontLightPreference front light 主開關與硬體輸出 gating

4 實際資料流

CLI / adb 路徑

settings put system
→
Settings.System
→
PowerManagerService observer
→
LogicalLight / HAL

screen_brightness 改變時直接套白光；screen_brightness_color 改變時會走 applyFrontLightColorLocked()；screen_front_light_mode=0 時會把白光與色溫都清成 0。

Settings UI 路徑

FrontLightPreference
→
putInt 到 Settings.System
→
直接 call PowerManager
→
硬體即時生效

白光 slider 走 setBacklightBrightness()；色溫 slider 走 setFrontlightBrightnessColor()；switch 關閉時會把 backlight 寫成 0。

5 跨元件同步機制

這套設計不是只有 Settings app 在控制，實際上是透過 Settings.System 當共享狀態，再由 system_server 消費。因此 CLI、UI、其他 service 都能共享同一組 key。

9 邊界與風險
  • 已確認到 LightsService / ILight 這層，但 tree 內沒有直接找到特定板子的 vendor light HAL / sysfs mapping。
  • 因此最底層 driver 節點目前只能合理推定是由 vendor HAL 將 BACKLIGHT 與 ATTENTION 對應到白光與色溫控制。
  • screen_brightness 共用標準系統亮度 key，是後續維護最容易踩到的耦合點。
  • LIGHT_ID_ATTENTION 被借來當色溫 channel，如果未來 attention light 要恢復原用途，會直接衝突。
