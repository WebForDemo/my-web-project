平台版本發佈
releasePlatformVersion

壓一版 hpnote_v1.0.0:把 system / vendor 兩棵樹同步到最新 → 更新發佈工具 → 解除 conda 環境 → 跑 releasePlatformVersion 上傳,system 與 vendor 各一次。

sync → 工具 → 環境 → release s 與 v 各一次 · upload = 1
01 / 06 一次搞懂

吃 6 個參數,做 5 件事

參數數量 ≠ 做幾件事。6 是你要「輸入」給它的東西;它拿到後內部做的是下面 5 個步驟。(6 參數版 vs 舊 4 參數版,只是把「網址、remote」從寫死改成由你輸入,做的事沒變。)

#內部步驟做什麼
1repo init + sync用基底 manifest 把樹切到該版本基底並同步
2createGlobalBranch在所有 repo 建版本 branch
3mergeVersionBranch合併版本分支
4updateVersionFilesbuildinfo_sys / vnd 寫入版本號並 commit
5createVersionManifest產生釘版本的 release manifest xml
s/v 各一次　版本名結尾**必須** _s 或 _v。system(b0)與 vendor(v0)是兩棵樹、各自發佈,所以跑兩次、得兩支 xml(`hpnote_v1.0.0_s.xml` / `hpnote_v1.0.0_v.xml`)。
02 / 06 前置 A · 同步到最新

system / vendor 都 sync + checkout 到最新

兩棵樹都要先 sync、切到對應 branch、reset 對齊遠端最新,確保 release 抓到的是最新內容。

# System 樹 cd /mnt/aosp2/android16_mtk_system_b0mp5_aiv repo sync repo forall -p -cv "git fetch hsrd7" # 不 fetch，下面 reset --hard 會找不到 remote ref repo forall -p -cv "git checkout -b hpnote_s" repo forall -p -cv "git checkout hpnote_s" repo forall -p -cv "git branch -u hsrd7/hpnote_s" repo forall -p -cv "git reset --hard hsrd7/hpnote_s" repo forall -p -cv "git pull"
# Vendor 樹 cd /mnt/aosp2/android16_mtk_vendor_v0mp5_aiv repo sync repo forall -p -cv "git fetch hsrd7" repo forall -p -cv "git checkout -b hpnote_v" repo forall -p -cv "git checkout hpnote_v" repo forall -p -cv "git branch -u hsrd7/hpnote_v" repo forall -p -cv "git reset --hard hsrd7/hpnote_v" repo forall -p -cv "git pull"
注意　reset --hard 會覆蓋各 repo 本地未推送的改動;確認你的修正都已 push 到對應遠端 branch 再做。
03 / 06 前置 B · 工具與環境

更新 build.sh + 解除 conda

1確認 build.sh 更新到 hs_master_s 最新版6 參數版的 releasePlatformVersion 只在 hs_master_s。把 ntx_addon/host_tools 對齊 master:
# 確認 /mnt/aosp2/android16_mtk_system_b0mp5_aiv/ntx_addon/host_tools/build.sh 為 hs_master_s 最新 cd /mnt/aosp2/android16_mtk_system_b0mp5_aiv/ntx_addon/host_tools git fetch hsrd7 git merge hsrd7/hs_master_s # fast-forward,拿到 6 參數版
2解除 anaconda,確認 unbuffer 正常conda 的 tclsh 沒有 Expect,會讓 unbuffer repo sync 失敗(can't find package Expect)。解除 conda 讓系統 tclsh 生效。
conda deactivate # 提示字元前面不再有 (base) unbuffer echo ok # 印出 ok = 環境正常
為什麼　① 6 參數工具只在 hs_master_s。② unbuffer 需要系統 tclsh 的 Expect;conda 環境會用到 anaconda 的 tclsh(無 Expect)導致 repo sync 失敗。
04 / 06 六個參數講解

releasePlatformVersion 的 6 個參數

#參數(以 _s 為例)意義
1hpnote_v1.0.0_s要開的 branch name(結尾 _s / _v)
2t-alps-release-b0.mp5-aiv-hpnote_s要下載的 manifest,選對應的;可到 .repo/manifests 查找
3ssh://ntx-sw@192.168.101.12/…/manifest.gitmanifest 來源,要用 ssh
4ntx_hsrd這個就是 remote(理論上還有對外跟台北的)
511 代表要上傳
6/mnt/aosp2/android16_mtk_system_b0mp5_aiv本地路徑,做 release 資訊蒐集
s / v 差異　只有第 1(版本名 _s/_v)、第 2(基底 manifest,b0/v0 對應 hpnote_s/hpnote_v)、第 6(路徑)不同;第 3、4、5 兩棵樹一樣。
05 / 06 完整流程

解除環境後,直接上傳

conda deactivate unbuffer echo ok cd /mnt/aosp2/android16_mtk_system_b0mp5_aiv
# system (_s) releasePlatformVersion hpnote_v1.0.0_s t-alps-release-b0.mp5-aiv-hpnote_s ssh://ntx-sw@192.168.101.12/mnt/android_fw_images/mtk/android16_mirror/platform/manifest.git ntx_hsrd 1 /mnt/aosp2/android16_mtk_system_b0mp5_aiv
# vendor (_v) releasePlatformVersion hpnote_v1.0.0_v t-alps-release-v0.mp5-aiv-hpnote_v ssh://ntx-sw@192.168.101.12/mnt/android_fw_images/mtk/android16_mirror/platform/manifest.git ntx_hsrd 1 /mnt/aosp2/android16_mtk_vendor_v0mp5_aiv
完成　兩條都成功 = 打版本完成,產出 hpnote_v1.0.0_s.xml + hpnote_v1.0.0_v.xml 兩支 release manifest。
06 / 06 重點整理

執行前掃一眼

  • 順序 — 兩棵樹 sync+checkout 到最新 → 更新 build.sh(hs_master_s)→ conda deactivate + unbuffer echo ok → 跑 release。
  • 第 3 參數 — 一律 ssh://…,不是 /mnt/android_fw_images/…(那是伺服器本機路徑,你電腦沒有)。
  • 第 2 參數(基底 manifest) — 選對應產品/分支的那支:system 用 …b0…hpnote_s、vendor 用 …v0…hpnote_v;不確定到 .repo/manifests 找。
  • 第 4 參數 remote — ntx_hsrd(對內);若要對外/台北另有對應 remote。
  • reset --hard — 會覆蓋未推送的本地改動,修正先 push。
  • conda — 沒解除會卡在 unbuffer: can't find package Expect。
產出　system 與 vendor 各一支 release manifest,共兩支,即完成版本發佈。
