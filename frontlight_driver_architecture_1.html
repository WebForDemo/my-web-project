前光控制底層架構指南 PowerManagerService (PMS) 鏈路

這份指南詳細解析了當你在畫面上拉動亮度拉桿，或是寫入 settings put system... 指令後，Android 系統是如何將這些數值一路穿透 Framework，最終抵達底層 PMIC 發光二極體的。

架構：System Server

▍2. 核心觀念：「偷天換日」的通道借用

這套架構最精采的地方，在於解決了 Android 原生架構的一個限制：Android 原生沒有「色溫」這個實體燈泡通道。

🤔 為什麼不自己新建一個通道？

如果要為 E-ink 專門創建一個全新的 LIGHT_ID_EINK_WARM 通道，工程師必須要修改：

  • hardware/interfaces/light/.../ILight.hal (底層 HIDL/AIDL 定義，動到這個可能導致 CTS 認證失敗)
  • LightsManager.java (Framework API)
  • LightsService.java (System Server 實作)
  • 以及底層 Vendor Tree 的 C++ HAL 實作。

這個牽扯層級太廣了。因此，Jungwen 採取了非常聰明且低風險的策略：借用系統中「原本就不會用到」的實體燈泡通道。

在 E-ink 閱讀器這種裝置上，並不需要像傳統手機那樣有「充電呼吸燈」或「未接來電閃爍燈」。因此，這些閒置的硬體通道就被徵收拿來對接前光的冷/暖 LED 燈珠了：

E-ink 實際功能 對應 Settings DB Key PMS 綁定的 LightsManager 原生通道
前光亮度 (冷光) screen_brightness LIGHT_ID_BACKLIGHT (最合理，原本就是螢幕背光)
前光色溫 (暖光) screen_brightness_color LIGHT_ID_ATTENTION (原本是通知注意力指示燈)
電源指示燈 (無, 系統接管) LIGHT_ID_BATTERY (原本是電池充電燈)

▍3. 攔截器 (SettingsObserver) 原理解析

當 UI 或實體按鍵修改資料庫後，SettingsObserver 會被觸發 (onChange() 回呼)。這裡是 PMS 的第二個重頭戲：開關與狀態的防呆判定。

💡 為什麼要有這個防呆機制？

如果系統目前是「前光關閉」 (Mode = 0) 的狀態，使用者如果用指令強行改變了 DB 中的亮度值 (例如 `settings put system screen_brightness 200`)，此時 底層燈泡絕對不能亮起來。

因此 Observer 中實作了強制的覆寫機制：如果是關閉狀態，無論亮度值為何，往底層傳的永遠是 `0`。

▍4. 寫入 HAL 服務

當判定完數值後，最終會透過 setFrontlightBrightnessColorInternal() 這樣的方法封裝，將一般整數轉換為底層 HAL 需要的浮點數，然後呼叫 mFrontLightColor.setBrightness()，資料就正式離開 Java 層，進入 C++ 的 Driver 層了。
