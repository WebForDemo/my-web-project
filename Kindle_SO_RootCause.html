1根因機制（一句話結論）

結論：系統內建的 libc++_shared.so 是舊版，缺少 Kindle 需要的 C++ 符號；又因它被列為「公開函式庫」而強制注入 App，蓋掉了 Kindle 自帶的完整版本，導致載入時找不到符號而閃退。

實際發生的崩潰

java.lang.UnsatisfiedLinkError: dlopen failed: cannot locate symbol
"_ZTTNSt6__ndk119basic_ostringstreamIcNS_11char_traitsIcEENS_9allocatorIcEEEE"
  referenced by ".../com.amazon.kindle-.../lib/arm/libKindleSDKFastMetricsJNI.so"

符號 demangle ＝ VTT for std::__ndk1::basic_ostringstream<char>

架構圖：為什麼會載到「錯的」函式庫

🟦 Kindle App 的 linker namespace
libKindleSDKFastMetricsJNI.so需要 → libc++_shared.so
libc++_shared.so（App 自帶 · 符號完整）/data/app/.../lib/arm/ · 含 ostringstream ✔
⤵
public.libraries 設定
強制改用系統版本
⬜ 系統 default namespace
libc++_shared.so（系統版 · 舊／缺符號）/system/lib/ · 無 ostringstream ✗
→ 找不到符號 → 閃退

關鍵：libc++_shared.so 被列在 public.libraries.android.txt，App namespace 因此被連到系統版本，而非自帶版本。

2關鍵證據與安全性驗證

① 不是 namespace 衝突，是「版本缺符號」

兩顆同屬 __ndk1 namespace，排除命名衝突；差別在系統版缺了整族 stringstream 符號。

項目系統版（舊）Kindle 版
檔案大小657,0001,301,936
__ndk1 符號18511912
匯出符號23042337
basic_ostringstream無 ✗有 ✔
Kindle 版相對系統版多出所需符號，但也少了 226 個（多為 compiler-rt builtin / 連結器符號 / libc++ debug 符號）。

② 替換是否安全？— 全面掃描

針對「Kindle 版缺少的 226 個符號」

掃描 image 內 1013 個 system / vendor 函式庫，檢查是否有人會用到這些符號：

結果：0 個函式庫 import 到任一缺失符號 → 替換對系統零影響。

③ 開機驗證

  • 換檔後重開機 正常開機，無 boot regression
  • 佐證「系統服務不依賴那些缺失符號」的靜態結論

工具：llvm-readelf -Ws 比對符號、readelf -d 查相依、/proc/<pid>/maps 驗實際載入來源。

3修正歷史：中途走錯路與更正

1
根因判斷（正確）確認為「系統舊版 libc++ 缺符號 ＋ public.libraries 強制注入」。方向正確。
2
改錯來源 ⚠️（中途錯誤）誤以為 /system/lib 那顆來自 prebuilts/ndk，替換了它並重 build、重燒機。
3
重燒後仍閃退同一個 ostringstream 符號錯誤再現 → 證明改的不是真正來源。
4
追 build 規則找到真來源查 ninja / PRODUCT_COPY_FILES，發現 /system/lib 其實是從 vendor prebuilt 複製來的。
5
更正並驗證還原 prebuilts/ndk 的誤改，改正確的 vendor 來源；rebuild 後確認目標檔已含符號。

錯誤的關鍵點：來源認錯

❌ 一開始改的（無效）
prebuilts/ndk/current/.../armeabi-v7a/libc++_shared.so
這顆是給「NDK 編譯」用的，不會進到 /system/lib。
✔ 真正要改的（有效）
vendor/aw/public/prebuild/lib/libc++_shared.so
透過 PRODUCT_COPY_FILES 複製成 /system/lib/libc++_shared.so。
教訓：同名 .so 在 AOSP 樹中有多份；改動前必須用 build graph（ninja / module-info）確認「該安裝檔的真正來源」，不能靠檔名猜。
